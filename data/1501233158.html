Introduction

This release note provides an overview of the improvements and new features introduced in the cadis 2025.5 release series (2025.5.0 – 2025.5.2).

Please refer to the cadis 2025.5 Change History for the complete list of all technical changes, improvements, and bug fixes.

Some of the functions described here may require additional licenses, and the availability of certain features can depend on your specific configuration, service cut level, and licensed options.

General Functionalities

New Features

cadisTRACK

With this release, CADIS introduces cadisTRACK, a brand-new solution for shipment tracking for your customers.

cadis now features a customer-facing tracking portal with configurable branding, a postcode security challenge, a milestone timeline (Registered → In Transit → At Delivery Hub → Out for Delivery → Delivered/Failed), ETA windows, and a live map. All mobile-friendly.

The system allows provides the ability to share tracking links externally, ensuring that stakeholders receive timely updates throughout the transport process. All status changes are transmitted immediately, which means that users see updates without delay. (RD-3735, RD-3763, RD-3764)

cadis “Online Help” overhaul

We have completely overhauled the cadis Online Help and moved it to a new more modern platform. The new Online help can be found under either CADIS Knowledge Base, the old URL (https://docs.cadis.cloud) or by pressing “F1” or selecting “Help” in cadis.

Mandatory cadisAPP Updates

A new mandatory update process has been introduced for cadisAPP. When a new version of the app is defined as required within cadisSTUDIO, users are prompted to update.

Logged in cadisAPP users will be warned five times with escalating severity once a mandatory update is set. The times for each warning level is configurable. The user can choose to dismiss the first four warnings.

At the final warning the app is blocked. The user must log out and update cadisAPP. This logout is possible even with open orders. After logging back in and completing the update, work can continue as normal.

This ensures that all users are always working with supported versions of the cadisAPP even if users remain logged in continuously and bypass the normal login checks. (RD-3688)

Improvements

User Management in cadisSTUDIO

The user overview in cadisSTUDIO has been enhanced with additional details, such as creation date, creator, last change date, last changed by, and last activity. This makes it easier for administrators to identify unused accounts and manage the account lifecycle more effectively. Furthermore, integration with OIDC has been improved, allowing first name, last name, email, phone number, and department to be retrieved from OIDC tokens and displayed in cadisSTUDIO. This ensures more complete and accurate user profiles. (RD-3971, RD-3972)

Improved Login Error Messaging

The error message displayed when the Azure login service is unavailable has been updated to be clearer for users. Instead of the previous message “Server error: User directory is not available”, users will now see “Login service not available. Please contact your IT administrator.” This helps reduce confusion and provides more actionable guidance. (RD-1916)

Geocoding and Map Enhancements

Several improvements have been made to geocoding and map functionality in cadisSTUDIO. Users can now place pins manually on maps in stop location details and collection creation dialogues, providing an alternative to automated geocoding. (RD-3047)

Improved version-update handling for cadisSTUDIO

cadisSTUDIO now handles version updates gracefully and transparently. When the server version changes, the client will automatically update itself, and if the server is offline or unreachable, the application now displays an offline page instead of an error.  (RD-3249 )

User Interface Refinements

The user experience across cadisSTUDIO has been improved in several areas.

  • The system no longer displays a loading spinner during routine background polling, ensuring that the interface remains responsive. (RD-3667)

  • Filter reset functionality now works even when the search bar contains text, so users no longer need to manually clear the field.  (RD-3989)

  • Quick filter text is now saved within filter views for consistent behaviour across sessions. (RD-4179)

  • Combo boxes in smaller masks have been adjusted to ensure that all values are visible and easy to select. (RD-4200)

  • The function menu accessed via the 3-dot button now has a consistent appearance and behavior throughout the application.

  • Submenus such as “Pin Column” are displayed more clearly (RD-3343)

  • the “Open” action has been renamed to “Details” for consistency (RD-3345)

  • STUDIO now cleanly distinguishes between “loading data” and “empty list” states. You’ll see a spinner when you actively load or refresh data (first load, filter change, refresh), but not during background polling. While still loading, the “No rows to show” text is hidden to avoid mixed signals. (RD-3667)

  • We have created some new more expressive icons for cadisSTUDIO. Below the new icons and their cadisCONTROL correlations. (RD-3595, RD-3306)

Pick Up and Delivery (PUD) and Terminal Handling (TH)

New Features

  • Unloading documentation workflows have been enhanced to support recording of “double-deck” loading , used and free loading meters  and the “loading unti full” flag. In addition, a photo documentation feature has been introduced for unloading. (RD-1552)

  • Outbound Route adjustment logic for transit depots in the case of shipment data forwarding by cadis (line haul: unloading list generated by cadis).

  • This feature provides business logic for a very specific use case. Assume a typical hub transport chain for shipment X with depots A-B-C where A is the collecting depot, C is the delivery depot and B is a fixed gateway for shipments from A to C. If the host system provides an outbound route of “B” for shipment X in depot A and the loading occurred on a line haul A-B with automatic creation of line haul unloading lists in enabled, then cadis will forward the shipment data unmodified from A to B. This did include the “Outbound Route” of the shipment (still “B”). If there was no further data update in “B” indicating the appropriate routing from “B-C” this would leave the outbound route of shipment X in Depot “B” still set to “B”. This may causing validation issues when trying to load B-C (wrong outbound route). The new logic provides that when a shipment arrives in the depot matching it’s previous outbound route (here “B”) cadis will now check if the shipment data also contains a value for “Destination depot” (which in this example should be “C”). If so it will automatically replace the “Outbound route” by the “Destination depot”. (RD-4091)

  • Harmonized “geocoding” indicators across cadisCONTROL and cadisSTUDIO.

  • Note: The icons may be colored depending on their placement in the GUI.

  • Meanings are (in this sequence): Geocoding Failed (red), GIS Geocoding Ok (green), Manual Geocoding (grey), Geocoding in Progress (black)

cadisSTUDIO feature focus

Numerous features previously only available in cadisCONTROL have been made available in cadisSTUDIO across shipments, dispatching, filtering, printing, and quick views in cadisSTUDIO. Highlights include:

  • The full range of shipment lifecycle actions can now be carried out directly in cadisSTUDIO, including dispatching, cancelling, changing execution dates and groups, adjusting priorities, creating partial shipments, and disqualifying or acknowledging shipments.

  • A wide variety of print reports are now available directly from the shipment overview.

  • Enhanced filtering options have been added, making it easier to locate shipments by execution date, sender, postal code, or product type.

  • Quick views have been improved to show the relevant details immediately when selecting a column.

  • The shipment overview now includes new icons, improved geocoding functionality, and better consistency between tables and maps.

  • In My Trips, sequence optimisation has been extended with the ability to display vehicle positions directly on the map, providing greater transparency during planning.

Please be aware that cadisSTUDIO feature development is still ongoing and not all features – large and small - of cadisCONTROL are already available. We are releasing a huge amount of features today, but we know that there are still plenty of gaps to the even bigger feature-set in cadisCONTROL. Bear with us as the scope of functionality available via cadisSTUDIO continues to expand rapidly with every release.

Bug Fixes

  • Improvements to the My Trips layout, ensuring correct display of departure numbers and action buttons. (RD-1996)

  • Shipment creation in cadisSTUDIO no longer requires delivery date and time fields. (RD-3541)

  • Package history now displays correctly when using the browser back button. (RD-3681)

  • Enhanced handling of courier shipments, with corrections to validation, data consistency, and geocoding. (RD-3963, RD-4201)

  • Fixed multiple edge cases where cancelled packages could leave behind invalid or “notAvailable” barcodes. Now, when a package is cancelled, its barcodes are reliably removed and dummy codes aren’t written into barcode tables. This prevents scanner confusion and downstream mismatches. (RD-3964, RD-4059, RD-4079, RD-4078)

  • Templates and line-haul routing have been stabilised, with empty addresses and depot assignments now processed correctly. (RD-4088, RD-4091)

  • Resolved an issue where partial shipments not created as expected when scanning packages across multiple MPs linked to different departures. (RD-4143)

  • Numerous interface corrections ensure consistency between shipment details and shipment overview screens. (RD-3977, RD-4092)

  • The width of Boolean columns in cadisCONTROL can now be adjusted individually across all tables, giving users more flexibility when viewing and organizing data. (RD-4186)

  • Disqualified shipments no longer appear “Out for delivery” (RD-4038)

  • Pickup template creation and updating can handle empty addresses correctly (RD-4088)

  • Trip template with updated vehicle or freight carrier can now be saved without server conflicts.  (RD-3980)

  • Fixed an issue where the name of “default freight carrier” in the vehicle master data got stuck after deleting the reference in the vehicle master data record and was reproduced in some screens (esp. the trip details). (RD-3997)

cadisCONNECT (API)

Improvements

  • Missing useCadisConnect* permissions have been added to several endpoints. These calls now require explicit authorisation, ensuring that proper permission checks are consistently enforced. (RD-2414)

  • Permission changes possible without reissuing API-tokens

  • API tokens (jwt format) can now be modified after being issued to add or remove permissions. Note: permissions encoded in the jwt are superseded by subsequent changes and may no longer reflect the actual permission granted. This behaviour is maintained for backward compatibility. (RD-4164)
