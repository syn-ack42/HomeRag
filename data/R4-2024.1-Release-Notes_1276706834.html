New Versioning Numbers and Release Cycle

With this release we have change our versioning scheme and our release cycle.

We will now publish a new cadis release every 9 weeks. More frequent releases mean more agile development, shorter delivery times and a smoother upgrade experience as upgrade steps are kept smaller.

Along with this new procedure we have changed the way we label our cadis R4 releases. Each release will consist of the year it was published and a counter of the release in the year. Even though it’s already late in the year “2024.1” is the first release in 2024 under the new naming scheme (our most recent previous release was R4.21)

In your cadis installation version information, you will find a third number after the release name e.g. “2024.1.1”. The last digit indicates a maintenance build of that particular release.

Creating partial shipments for packages in a master pallet via cadisAPP

License requirement: LPP-MPL

This feature is an extension of the functionality to create partial shipments via the cadisAPP.

Create partial shipment via cadisAPP

This functionality allows workers on the terminal to conveniently create “partial shipments” for packages that were left behind on a cross-dock. When such a package is scanned with cadisAPP the user is asked if they want to create a partial shipment. Up to now this was inconvenient for packages already contained in master pallets (MP) as they are not individually scanned. Users would have to open the MP and scan individual packages to create a partial shipment.

With this release cadisAPP users only needs to scan the MP barcode. They are then asked whether they want to create partial shipments for the MP content where needed.

Product information and Release Notes now available to all cadis users via the online help

We have reconfigured our online help to allow all signed in cadis users access to the “Product Information” section, which includes these release notes.

As long as you have a valid login to a cadis installation (>= R4.18) you can conveniently access this website by using the “Help” link or pressing “F1” in either cadisCONTROL or cadisSTUDIO.  See How to Access Online Help and Release Notes for details.

Other Improvements and Bugfixes

  • Fixed the issue where some browser opened up the password auto-fill dialogue inappropriate locations (such as the “Authentication type” menu in the mobile access screen)

  • Fixed an issue in the Koovea temperature control integration where delayed data was not considered for the temperature curve

  • Improved flexibility in storing perspectives for cadisSTUDIO tables. Different aspects of the views can now be stored separately: Columns (column order, width, group-by, sort), Filters and Charts. For each of these three, users can store multiple named views. Users are free to combine these stored settings freely (e.g. switch through different filters while leaving the displayed common set.)Also the generic table display settings (row height, visibility of the status bar, column-control menu) are not part of the stored view anymore but will always return to the last state selected.

  • Info Scan: The feature has been moved to the main menu of cadisSTUDIO.

  • The scanner is now active during cadisAPP login in “Mobile Device” and “Driver” mobile access mode. That way a vehicle or device ID can be scanned instead of manual entry.

  • To increase robustness, barcodes with leading or trailing whitespace are now trimmed before using them. Consequently, barcodes may not intentionally contain leading or trailing blanks, tabulators, or line feeds. If they do (both in the printed barcode and in the shipment data), cadisAPP may not find the corresponding package.

Pre-set packages missed during loading as “missing” in delivery

License requirement: PUD-LDR or PUD-PLD

If enabled in configuration (Trip processing => Mark not loaded packages as missing; default: false) packages that were not loaded (“missing” or “not loaded” after loading is closed) are pre-set to status “Missing” during the delivery. This way, drivers can immediately see that they may be missing a package when starting the delivery and they don’t have to set this status manually (they can, of course, change that status should the package turn up after all).

Loading Equipment Exchange: Optionally use quantity recorded at loading instead of shipment data

License requirement: PUD-LDR or PUD-PLD

It’s now possible to use the number of loading equipment recorded during loading for the expected loading equipment quantities during delivery in cadisAPP. If enabled in configuration (Trip processing => Use loading equipment count from loading), the number of loading equipment recorded as actual in loading replaces the “expected amount” in the shipment data.

Configurable Stop Location matching radius

The radius for matching imported addresses to Stop Locations (master data) is now configurable. The default is 3 meters (which was the previous hard-coded setting).

Background: when shipment data is imported, cadis tries to match the relevant addresses (delivery address for deliveries, collection address for collections) to existing stop location master data records in order to amend the shipment data. A Stop Location is matched if either:

  • the Address-ID is given and references a stop location master data record

  • no Address-ID is given and the “Name1” of the addresses matches either the “Name1” of the Stop location address or one of the Alternative Names of the stop location record AND the geo-location of the Stop Location is within the matching radius of geo-location of the shipment address

Other Improvements and Bugfixes

  • The summation of the “Utilization” columns at the bottom of the trip table in the cadisCONTROL Trip monitor has been removed as it did not make sense and could even be misleading. (Background: The “Utilization” of each individual trip is calculated and displayed as the maximum of weight and pallet place utilization. Making a sum of those lines means adding different measuring units to each other.)

  • Fixed an issue where sequence-optimizing a trip more than 3 days in the future using PTV 2 failed with an error message.

  • Fixed an issue where status/sub-status configured as “not selectable by user” could still be selected on the cadisAPP

  • Collection template creation: when creating a collection template using a “Stop Location” for the shipper address, the suggested collection days in the schedule are now pre-set from the stop location master data record, i.e. days where the location is marked as “Closed” are now pre-set to “deactivated” for generation of collection orders. (Note: this is only applied when creating a new collection template. Once a template has been created changing the stop location maser data does not affect the template.)

  • Added a column “Consignee – Street 2” in Shipments overview, Trips (Tabs “Stops”, “To be dispatched”), Loading details (Tabs “Shipment”, “Stops”)

  • Add shipment via myTrips on the cadisAPP: The function now only provides actually dispatchable shipments (previously all shipments found using the search reference were displayed although usually only one of them could actually be dispatched).

  • Sequence optimization: in the new cadisSTUDIO dialogue, the handling time per stop is now editable and the final few columns that were available in the cadisCONTROL but not yet in cadisSTUDIO have been added.

  • Dangerous goods document: if the trip has more than 1000 dangerous goods points, the “The vehicle must be labelled” section and the “ADR certificate and identification…” sections are automatically checked.

  • Proof of delivery report (ePOD): the Global Tracking Number can now be printed in the pdf (can be controlled via configuration, default is “off”).

  • Proof of delivery report (ePOD): The GEO-Coordinates can now be printed in the pdf (can be controlled via configuration, default is “off”).

Configurable Loading Equipment actual amount pre-set in Loading/Unloading/Inventory

cadis now features a configurable loading equipment preset mode for loading, unloading and inventory. This setting controls how the “actual amount” is pre-set when the loading equipment recording screen is shown on cadisAPP.

The available modes are:

  • ‘ZERO’: The value of the “Actual” input will be pre-set to “0”. The dialogue can be saved and closed directly.

  • ‘ONE_TO_ONE’: The value of the “Actual” input will be pre-set to the “Expected” value. The dialogue can be saved and closed directly.

  • ‘NONE’: The value of the “Actual” input will be empty. The dialogue cannot be closed without input.

This configuration can be set separately for Loading, Unloading and Inventory. Within loading and unloading the setting can be differentiated between traffic type (local traffic or line haul) and user type (driver or terminal worker)

cadisCONNECT

cadisCONNECT provides an API layer for programmatic access to cadis functionality.

Depending on the use case, APIs may be designed for:

  • Integration use cases: retrieve detailed information about specific business objects (e.g., the ETA for the delivery of a certain shipment) or perform actions in cadis (e.g., post a note to a shipment)

  • Mass data load use cases: perform cyclic data retrieval for use in business intelligence solutions.

Please check out our white paper for suggestion how to handle business intelligence mass data download use cases: Introduction to cadisCONNECT for Business Intelligence

The API documentation for your current cadis release version is also directly available on you own cadis domain under the same path (https://<your.cadis.domain>/cadis-rest-public-doc/). Please sign into cadisSTUDIO first to authenticate before accessing the documentation URL. Please note that all API specs can be read on the public documentation, but to actually use the API requires the appropriate license to be activated and for the user/token to have the relevant permissions assigned.

Please note that we are currently in BETA with active development ongoing.

  • available APIs don’t yet cover the full range of cadis

  • APIs are subject to change

  • we appreciate feedback!

Product Lifecycle Announcements

Deprecations

PTV xServer 1 support

PTV has announced that support for xServer 1 will be finally discontinued in 2026 at the latest.

As cadis R4 already fully supports xServer 2, we plan to migrate all remaining xServer 1 customer to xServer 2 and discontinue xServer version 1 support in the first half of 2025. As operation of xServer 2 is more complex demanding than xServer 1, cadis will not support on premise installations of xServer 2. Instead, all xServer functionality, even for our on premise customers, will be provided via our cadis SaaS environment (Note: if required, cadis itself can remain on premise while xServer functionality is provided by the cloud).

Should this affect you, our cadis representative will proactively contact you in a timely manner.

Discontinuations

Android 7 no longer supported

In line with our Android version policy Android Version 7 (and lower) are no longer supported on cadisAPP. This is necessary, due to changes in Google Play store API level policies.

JMS support removed

As announced at the end of 2023, JMS support is now discontinued from our message broker.

Data-lake and legacy data lake API removed

As announced in the R4.21 release notes the legacy cadisAnalytics API (Swagger API Definition “Otms”) and the underlying data lake have now been removed and are no longer available in 2024.1
