New functions in cadis Release 4.16

Logistic Process Platform (LPP) – General functionality

If not stated otherwise new features are generally available in all basic service cut levels (LPP-ENT or LPP Cluster) but accessibility or utility may vary with the limitations of the service cut and your individual usage of cadis. In some cases, availability of information and/or functions in a feature may depend on service cut and other licensed options.

User management levels

This is major upgrade to our role/permission management for customers using the cadis internal user management. There are situations in which new login credentials may need to be issued to individual users. Our service managers will support you proactively before, during and after the cadis upgrade.

IT security is of increasing importance and guidelines and best practices are evolving all the time. CADIS takes IT security very seriously and we are constantly evolving and modernizing our security module to meet current best practices for authentication (is a user known?) and authorization (what is the user allowed to do?).

In this major release we have introduced changes that enable us to address important IT security principles like “Principle of Least Privilege” and „Principle of Separation of Duties “.

The immediate changes are:

  • all users are now global and unique to a tenant with depot assignment being handled in the user account.

  • we introduce a separation between operational permission for business functions and administration permissions

  • we introduce an administration level hierarchy to minimize the permissions assigned to a user

These changes mostly affect customers using the cadis built-in user management. If you are using an external user directory via LDAP or OpenID Single Sign On you will notice little if any change as these systems generally operate under these paradigms already.

Unique user logins, multi branch logins

Usernames now have to be unique across all depots of a tenant and each user may now be assigned to multiple depots. This replaces the old practice of creating a user with the same login name on each depot they need access to.

User may also be assigned different roles in different depots. In the example screenshot below. The user “cadis” may work as a customer service agent in DE-FRA and DE-NUE but may only dispatch local traffic in DE-FRA.

In this concept usernames must be unique across all depots. Before cadis is updated to a release >= R4.16.3 our service department will check if there are users with the same login name in different depots. In case there are they will ask you to check if the users are distinct or represent the same person.

  • If a user is distinct you will need to change the “Login name” of the user(s) to make them distinct also (e.g. by appending or prepending the depot). This will not change the password, but the user will of course have to be notified of their new login name.

  • If two or more users in fact do represent the same person you will have to make sure that the leading entry has the most complete data. During the version update cadis will automatically merge users with the same login name to the leading entry and provide those users with access and roles for all the depots they previously held. All user data including the password will be that of the leading entry. Our support will provide you with information on what the leading entry is (in most cases it will be the first one created under that login name, but there are edge cases).

Management levels

Each user can now be assigned a “management level” to control administration functions.

Management levels are

  • None: most users should have this level. They can perform business functions only but can’t change create or edit users or roles.

  • System: root access. There should be very few users with this level. A “system” user can perform every administrative function in every depot. They can edit all users and roles and change even the most critical system parameters. (SaaS clients cannot be issues with “System” administrator users.)

  • Users globally: can manage users in all depots and can edit roles. E.g. admin in the central IT department

  • User locally: can manage users in the selected depots (and only there) but can’t edit roles. E.g. a key user for a branch location who can create or change users for their local depot.

Users with a management level other than “System” still need business roles assigned to execute business functions (but they can assign them to themselves). A management level user can, however, never assign a management level they do not possess for themself.

During the update cadis will automatically assign management levels in line with the users existing permissions in cadis. As it is not uncommon for users to be over-privileged in cadis it is recommended to review the assignment afterwards and take advantage of the improved ability to implement the “Principle of Least Privilege” – i.e. users should only have the privileges they need to do their job. This is especially important for admin privileges.

Android 13 support

The cadis APP now supports Android 13.

Note: Android 13 introduces a new permission for “system notifications”. If that permission is not granted by the user (or via MDM) cadis won’t be allowed to show notifications but will work normally otherwise.

cadis STUDIO UX improvements

We did several cadisSTUDIO user experience improvements, especially for smaller devices (e.g. when used through the cadis APP.

  • When opened on a small device (e.g. inside the cadis APP on scanner) the headers are now displayed as icons only

  • Multi-tab pages can now be “swiped” on touch-screen devices

Customers and StopLocation Master Data in cadisSTUDIO

The migration of StopLocation and Customer master data to cadisSTUDIO is now complete.

Both data types can now be viewed, created and edited via cadisTUDIO.

We have added the possibility to add StopLocations to Customer data records via the master data gui. Linking stop locations to Customers supports the user during order data entry (e.g. entry of collections) by filtering the possible stop locations.

Default fallback now consistently in English

In cadisCONTROL and some print product the default fallback language if a resource key is missing a translation in the current language was still German. These features now also default to English.

Improvements to mobile device management

The mobile device management screen now shows the last logged in user (which might be a vehicle or device name).

Old log files are now kept after new ones are requested and remain available for download.

BETA: Add and change cadis APP translations at runtime

Many translations in cadisAPP can now be added or changed at runtime by uploading resources to the cadisSERVER. Some restrictions still apply e.g. for status/substatus values, we are working on that in R4.17.

This feature is in BETA so changes in resources must be tested and applied by a qualified CADIS system integrator.

Pick Up and Delivery (PUD)

If not stated otherwise new features are generally available in all basic service cut levels (PUD-DRV, PUD-LDR, PUD-PLD) but accessibility or utility may vary with the limitations of the service cut and your individual usage of cadis. In some cases, availability of information and/or functions in a feature may depend on service cut and other licensed options.

Dispatch confirmation dialog “Dispatch and Close“ behaviour

Clicking the “Dispatch and close“ button in the dispatch confirmation dialog will now dispatch all shipments in the category “Dispatchable without warning”.

Please note that it will not automatically dispatch shipments with dispatch warnings as these need to be overruled.

Enhance PUD-Drive (trip execution only) with cadis planning

License requirement: PUD-DPL

With this enhancement package you can upgrade a PUD-DRV service cut to allow user in cadis to modify and enhance trips by using cadis planning functionality. This package includes intelligent logic to avoid conflicts with the host system when the same trip (or set of trips for a vehicle on a given day) is modified both in cadis an in the planning system.

Use-Cases:

  • add collections to a trip received from the planning system:

    • with PUD-DPL you can enter pick-up shipments in cadis or auto-generate them from templates (otherwise only available in full “PUD-PLD: Plan Load and Drive” license)

    • Dispatch collection to trips received from the planning system via interface

    • cadis will intelligently integrate trip-updates from the host system with dispatching decisions in cadis

  • Pre-plan collection trips and merge them with trips from the interface

    • create a trip in cadis and pre-plan collections (e.g. for the next day)

    • configure cadis to automatically merge trips received from the planning system with pre-planned trips in cadis (for a given vehicle on a given day)

  • Second trip merging (second wave handling)

    • configure cadis to automatically merge remaining unprocessed collection or delivery orders into a second trip received from the planning system. Optionally cadis can also merge collection or delivery orders with non-final issues (those where a driver may attempt a second attempt later) in addition to the not processed ones.

Trip monitor row display Shipment/Trip properties

Display of the “Trip properties“ and “Shipment properties“ columns in the “Trips” and “Stops” table (respectively” of the “Trip overview” causes the rows in the table to become double height. If you do not like the double high rows but you do like the information displayed in these columns there are now single line alternatives available.

  • “Shipment properties“ (first line of the classic view) and “Vehicle properties” (second line of the classic view) in the list of trips

  • “Shipment properties” in the table of stops (the classic view column has been renamed “Shipment properties multiline “

If you prefer you can now enable these new columns and disable the classic multiline columns.

Auto-Dispatch cut-off configurable

License Requirement: PUD-PLD

The “Auto-Dispatch” feature can now be configured to allow auto-dispatching depending on the status of the departure.

When the auto-dispatching is triggered either manually or via the rule engine output parameter cadis will dispatch matching shipments to departures that are in one the configured status.

Selectable status:

  • Created

  • Released for loading

  • Loading in progress

  • Waiting for confirmation of completion (driver waiting for confirmation)

  • Finished (but not yet closed, by terminal worker)

Mobile printer support in collection and delivery

License Requirement: Individual (available in PUD). Validation and integration of printer hardware and label design required.

Important: This feature is not “plug and play” due to the wide variety of hardware and label formats and layouts.

  • Hardware customization required.
    Mobile lable printer hardware varies widely. Evaluation of the intended equipment is required and the need for integration of proprietary drivers is likely.

  • Layout customization required.
    Label layouts require individual design and data mapping.

cadis adds capabilities for printing on mobile lable printers (usually Bluetooth) during collection and delivery.

Supported use-cases:

  • print collection information label, e.g. identification barcode for packages (printable for all collection orders)

  • print collection receipt (printable for completed collections)

  • print delivery card, e.g. notification card or receipt (layout configurable per delivery order status/sub-status)

Adding packages to shipments via cadis APP

With this new feature users in cadis can now record additional packages for shipments during loading, unloading and inventory.

If the feature is enabled the users

  • may choose the “Add package to shipment“ function from the shipment context menu

  • when scanning a barcode for which no data is found directly after scanning a known package, i.e. while they are in the scan response screen, they will be asked if they want to record the package as a unknow surplus or add it to the shipment.

Shipments with packages added by a cadis APP user will be marked in various cadis overviews and the number of packages added is displayed where appropriate.

The feature can be enabled separately for loading, unloading and inventory and, in the case of loading unloading, for the different process types (LineHaul and LocalTraffic) via runtime configuration.

Allow loading of unknown packages (“noEDI” loading lists)

License Requirement: PUD_PLD or TH_PCS

cadis can now be configured to allow creation of “no EDI” loading lists that all loading of unknown packages.

If the feature is enabled via runtime configuration users are allowed to created loading lists with the “NoEDI“ flag enabled. Loading lists with this flag allow loading of packages without shipment data via the cadis APP. NoEDI loading lists are always dynamic, i.e. they allow adding shipments via loading scan.

The feature can be enabled for local traffic and line haul loading separately.

MasterPallets with packages without data (“noEDI” master pallet)

License requirement: LPP-MPL

If enabled (see configuration) packages without shipment data can be added to master pallets. The user must select the “packages without data allowed” option when creating the master pallet. No MP content validation checks are performed when adding packages without data (as there is no data to validate). On further processing of the MP (e.g. loading, dispatching etc.) all validation checks relevant to that business process are observed. For example, if loading of unknown packages is forbidden (as is the default) then an actual loading scan of such a MP will be denied.

Note when used in combination with the generation of line haul unloading lists by cadis. cadis does not include “unknown packages” (packages without shipment data) when generating an unloading list for the recipient depot of a line haul. As a consequence, cadis also does not include MPs that allow unknown packages in these unloading lists.

Dangerous goods details in info scan

Detailed dangerous goods information is now available via the shipment details / info scan screens. If a shipment contains dangerous goods, the pertinent hazard label is displayed. Clicking on the label will show the full details for the dangerous goods item in question.

Info Scan / Shipment detail improvements

We took care of a lot of details to enhance user experienced (and fixed some oversights)

  • Both the actual and planned storing location are now displayed prominently in the “Quick view” tab of the InfoScan. If the expected location is set and the current actual location differ a warning marker is shown.

  • The handling instruction master data text is now shown properly together with the instruction code

  • Addresses have been moved to their own tab for quick access while not taking up screen space in the info section

  • Context sensitive Handling instructions: When the shipment details are opened from an ongoing process only the handling instructions relevant to that process are shown. e.g.: open shipment details during an “Unloading” and only the Instructions with the types “Unloading”, “Scan” and “General” are visible. The “Info-Scan” continues to show all instructions.

  • Removed the duplicate display of Shipment IDs in the info section. Both the “HostID/OrderNumber” and the GlobalTrackingNumber are already displayed in the header. No need to show them again.

Use hardware scanner in cadisAPP Info-Scan

License requirement: DPT-ISC-ADV for the storage location feature

When using “Info Scan” with barcode scanner hardware it is now possible to scan the next barcode directly in the Info-Scan result screen without returning to the “Scan” dialog.

Supported use cases:

  • Scan another package, a new info scan will be executed directly

  • Scan a location function barcode. The scanned location will be assigned to the package.

Info Scan: Free-text storing location in set actual location

License requirement: DPT-ISC-ADV

Free text storage locations (storage locations not defined in the master data, also called “special locations”) can now be set via the “Set storage location” feature of the info-scan.

Confirmation barcode process for loading equipment issues in loading

License requirement: TH-LME

cadis can now be configured to require the scan of a confirmation barcode before a deviation in loading equipment amounts can be recorded.

A loading equipment deviation is:

  • the actual amount is not equal the expected amount

  • any not exchangeable loading equipment

  • in case of loading equipment with id’s: the number of barcodes recorded is not equal to the “actual” amount entered

Deviations have to be validated whenever the “loading equipment” documentation screen on shipment or on loading list level is saved. (See also notes at the bottom of this section.)

In this workflow the user must scan a specific barcode before the cadis APP allows the user to continue to store the loading equipment deviation. The confirmation barcode should only be in the possession of authorized supervisors allowing them to take note of the problem on site.

Whether a loading equipment deviation requires barcode confirmation, and the content of the confirmation barcode is configured via runtime configuration.

 Please note:

  • the confirmation process is currently only available for loading. Configurations for other processes are ineffective

  • For recording on shipment level, it is recommended to configure the cadis APP to request the documentation of loading equipment on scan of the last package of the shipment.
    The confirmation dialog is not triggered when adding loading equipment via function barcode scan. This functionality is independent from the confirmation and can be used to incrementally add loading equipment which then is confirmed and validated on the last package scan.

Always show planned storage location in scan response

Both the actual and the expected storing position are now always displayed prominently in the scan response for unloading, inventory and loading. If the actual storing position does not match the expected location, it is indicated with an attention symbol (see screenshot).

Naming of unknown packages

The fixed designation “ÜZ” for surplus packages has been abolished.

The prefix for unknown packages is now configurable (default “SP_”).

Changes:

  • Addresses (Consignor, Consignee, Shipper):

    • Name1 is filled with the configured prefix, operation name and date of the first scan/occurrence of the unknown package. e.g. “SP_LOCAL TRAFFIC 111”,2023-02-06

    • StreetAndNumber is filed with the prefix and the full timestamp of the first occurrence e.g. “SP_2023-02-10T17:51:23.572Z”

    • Since the inventory has no name, a composition of area/place and type is formed.

  • cadisAPP: Shipment lists in the processes / master pallet: “Unknow Packages” renamed to “Single Packages”.
    That’s because it is not always an unknown package, it could be a package of a shipment which was cancelled afterwards.

  • Shipment data: The package barcode with prefix is used as the “Shipment Number” (OrderId/HostId) and as the GTN-Number (GlobalTrackingNumber) (ATTENTION: the HostID is now also filled not only GTN)

  • The orange marking of the “shipment number” field is still present in loading/unloading/inventory details.

  • A history entry is written for the package.

Arrival board details for products and priorities

License requirement: MWS-ARB

The “Arrival Board” has been enhanced with a “Details” widget showing the number of Product and Priority shipments.

Configuration of list search return limit

The maximum number of results returned on a search for loading/unloading/inventory lists can now be configured. The default is 50. If you make a lot of lists available to your terminal workers, you can now have this value changed to avoid “too many results” error messages.

SMS with web link

License Requirement: YM-DCT, full functionality only in YM-PLC. Lower license levels are limited to the available functionality (e.g. no shunting orders)

Whenever a truck driver receives an SMS from the yard manager, there’s now an additional link available in the SMS to show a graphical notation of the SMS text and to show a graphics of the yard. The cadis system must be reachable via the internet and the url path for the cadis “tiny-url” service (“/go”) must be added to reverse proxy configuration.

cadisCONNECT (BETA)

cadisCONNECT provides an API layer for programmatic access to cadis functionality.

Depending on the use case, APIs may be designed to

  • integration use cases: retrieve detailed information about specific business objects (e.g. the ETA for the delivery of a certain shipment) or perform actions in cadis (e.g. post a note to a shipment)

  • Mass data load: perform cyclic data retrieval for use in business intelligence solutions

The API documentation for the current product major release is available via https://demo.cadis.cloud/cadis-rest-public-doc/ with a "Swagger UI" fronted. The url is protected by a login. If you are interested in cadisCONNECT please contact your project-, service- or account-manager for login data and a brief introduction by one of our consultants.

The API documentation for your current cadis release version is also directly available on you own cadis domain under the same path (https://<your.cadis.domain>/cadis-rest-public-doc/). Please sign into cadisSTUDIO first to authenticate before accessing the documentation url. Please note that all api specs can be read on the public documentation, but to actually use the api requires the appropriate license to be activated and for the user/token to have the relevant permissions assigned.

Please note that we are currently in BETA while active development ongoing.

  • available api’s don’t yet cover the full range of cadis

  • api’s are subject to change

  • we appreciate feedback!

We will remain in BETA for a while as we are adding new functionality with every cadis release.

Refactoring of “Items” API

As announced with R4.16 we refactored the function signatures of the “Item” api . Specifically, parameters that are currently in the “query” section of the api have been move to either the “path” or the POST request body to be more compliant with common REST practices.

Other important changes:

  • GET /{branch}/items/details

    • isLabellingObligation removed from response (this is an internal value exposed by mistake)

    • hasDangerousGoods added (this is what should have been there instead of labelling obligation)

    • dangerousGoodsPoints data type changed from “integer“ to “number” (i.e. floating point, because technically calculated dangerous goods points can carry fractions) Note that this applies to the sum on shipment level and the individual dangerous goods lines

    • ItemPackageIdentificationDetailsDto (package details)

      • actualStoringPosition added

      • hostID and cadisID added

      • barcodeContent renamed to hasAdditionalBarcodes (it’s still “true” if the package has additional/alternative barcodes

Appendix 1: Interface updates

This chapter covers changes to the message based cadis interfaces as well as new message based interfaces. Please check the sections on “cadisCONNECT” for updates to the cadis APIs.

cTOrderstate export for partial shipments

cTOrderState will now also be exported for partial shipments in cadis. The “OrderID“ of the partial shipment will be the ID created in cadis for the PartialShipment.

Please see also the R4.15 Release notes Appendix 1 section: “cTLoadingStatus/cTUnoadingStatus/cTInventoryStatus addition of partial shipment information“. The “OrderID” in cTOrderstate is the same ID as the PartialShipment “HostID” used in these status exports.

Default Barcode of containing MasterPallet added to cTLoadingStatus

The “MasterPalletReference” block appended to the loading status of a package if it is contained in a master pallet has been extended to include the DefaultBarcode of the master pallet.

Note: This addition is for human reader convenience only. For a technical reference to the master pallet please use the HostID to ensure uniqueness.

CadisXmlVersion >= 5.2.50 has to be enabled to include this value.

Appendix 2: New run time configuration options

Please contact our service organization to arrange for a consulting meeting to improve your workflows by leveraging these new options.

Availability and scope of configuration options depends on the underlying functionality being licensed.

New/changed PUD run time configs

  • auto dispatching departure states
    Configure the status of a loading lists that permit the auto-dispatching functionality to dispatch shipments onto the list (see feature description “Auto dispatch cut-off configurable”)

  • Runsheet variant (runsheetVariant)
    Replaces “runsheetIncludeBarcodelist“. Select the run sheet print document variant.
    Standard variants (RS_STANDARD_001: without barcode list RS_STANDARD_002: with barcode list)

  • Ignore cTTour sequence from host
    Ignore stop sequence changes in cTTour updates from the host

  • Pickup Receipt Template / Package Label Template / Delivery card
    see. feature “Mobile printer support in collection and delivery”

New/changed TH run time configs

  • inventory/loading/unloading search limit
    the maximum number of search result returned to cadis APP when searching for a loading/unloading/inventory list

  • Approval barcode for package deviations (in loading)
    Validation barcode for package deviations (see R4.15 feature “Confirmation barcode process for package issues in loading”)
    Configurable separately by traffic type LineHaul/LocalTraffic and user type TerminalWorker/Driver
    As of R4.16 this switch is only effective for loading (it does nothing for unloading and inventory)

  • Approval barcode for loading equipment deviations (in loading)
    Validation barcode for loading equipment (see feature “Confirmation barcode process for loading equipment issues in loading”)
    Configurable separately by traffic type LineHaul/LocalTraffic and user type TerminalWorker/Driver
    As of R4.16 this switch is only effective for loading (it does nothing for unloading and inventory)

  • Add package to shipment allowed (in loading, unloading, inventory)
    Adding packages to shipments is allowed (see feature “Adding packages to shipments via cadis APP“)
    Configurable separately by traffic type LineHaul/LocalTraffic (for loading, unloading) and user type TerminalWorker/Driver (for loading)

  • prefix for single package creation
    The Barcode prefix for creating unknown packages (see “Naming of unknown packages“)

  • Allow unknown package in loading (createNoEdiPackageAllowed)
    Allow the recording of new “unknown packages” (packages without shipment data) during the loading process. (Configurable separately by traffic type LineHaul/LocalTraffic and user type TerminalWorker/Driver)

  • Enable “no EDI” master pallet
    Allow creation of “no EDI” master pallets (see feature “MasterPallets with packages without data (“noEDI” master pallet)“)
