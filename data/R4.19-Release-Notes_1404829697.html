File based data exchange with cadis SaaS in AWS (leveraging S3 buckets)

cadis now supports AWS S3 buckets for input and output of interface messages.

Leveraging S3 buckets we can provide you with:

  • sFTP server data exchange with on premise systems

  • direct S3 bucket data exchange (AWS internally or via S3 clients)

  • Cost-effective long-term storage of interface data (e.g. backing up all status messages, images etc.)

Internationalized Display of Vehicle Types in cadisAPP 

We have enhanced the “Vehicle type” field display in cadisAPP. With this update, the “Vehicle Type Name” from the “Vehicle Type” master data will now be shown (not only the ID).
If no “Vehicle Type Name” is set cadis will continue to display the ID.

In addition, we have introduced translation support for the “Vehicle Type Name”. cadis displays the translated vehicle type names based on the users’ language settings.

You can translate the “Vehicle Type Name” when using the master data import interface. The cTVehiclesTypes master data interface has been extended accordingly with a new field for the translated versions of the “Vehicle Type Name”.

Please note that this translation feature is currently not available when managing the vehicle type master data via cadisSTUDIO GUI.

Message channels naming of  “my” user Channel to Supervisors 

Until now all user channels, including “my own” were named after the person: User account name, First name, Last name.
This is fine for all other user channels, but confusing for my own one: Here “I” communicate to the group of supervisors.

It is now visualized as a separate from the other “user channels” and named “Supervisors”.

Integration of Data for Unknown Packages in Master Pallet

The wording “unknown package” refers to a package that has been registered (scanned) in cadis but for which no shipment data was available at the time. These packages are recorded and tracked in cadis and can be processed in unloading, loading or inventory (depending on configuration). When shipment data is received later which includes the package it is matched via the barcode and the “unknown package” is integrated with the shipment data.

This data integration now also propagates to a master pallet if the “unknown package” has been assigned to it. Once the data is integrated all subsequent validations and checks will use the shipment data to prevent errors (like loading to the wrong truck) using the best data available at the time.

Grouping stop locations with alternative spelling

License requirement: PUD-PLD

The (correct) grouping of deliveries/pick-ups to stops within a tour has multiple benefits: the tour calculation is improved because stop durations are calculated appropriately, dispatching is easier because there are fewer entries in the stop list and mobile execution for the driver is easier as the list of stops is displayed concisely and the optimized stop workflows are in effect.

Automatic consolidation of stops is the goal.

Stops on a trip are automatically consolidated in cadis based on the geo-coordinate and the address. If, however, shipments with addresses with different spellings are imported from the host system, this can lead to several stops in one trip. This new feature allows the maintenance of alternative spellings of the name in the stop locations master data which significantly improves the chances of collections and deliveries being automatically grouped into stops correctly – especially as the system learns new name variants over time. 

See Grouping stop locations with alternative spelling – CADIS Online Help for information on this feature.

Improved configurability for automatic order status selection based on package level deviations

When a collection or delivery is processed on package level (i.e. the driver sets a status for each package individually) cadis automatically derives the overall order status from the individual packages status.

Siple example: If a shipment has 2 packages and both are delivered “OK” the overall-status of the delivery order is set to “Delivered OK”. If, however, one package was OK and the other missing the default in cadis was to set the overall status to “Delivered – with deviation”. The same overall status was applied if one package was “OK” and the other “damaged”. It is now possible to configure the derived sub-status with more detail. In the above example cadis could now automatically return “Delivered – Partial delivery” and “Delivered – with damage” respectively.

This configuration can be set per workflow variant.

The default for update installations is to keep the old setting so as not to affect integration with 3rd party systems (for brand new projects an improved default configuration is applied). If you would like to take advantage of the new capabilities, please contact our service organization.

Background Sequence Optimization on Trip Change

License Requirement: PUD-PLD, PUD-TRO

If this new feature is enabled cadis will automatically try to re-optimize the sequence of trips if changes are made to the trip. This will help dispatcher to quickly see the results of their dispatching actions and always have “optimal” trips while working on their detail dispatching. 

To avoid unnecessarily optimizing incomplete trips the automatic optimization will only be done on trips that were sequence-optimized at least once (e.g. manually when the trip is sufficiently planned). Whenever the tip is affected by a dispatching change (e.g. adding or removing a shipment or a change to shipment data) cadis will attempt a sequence optimization in the background.

Automatic optimization is disabled once the vehicle has left the depot to avoid sending frequent updates to the driver.

Create Partial Shipments via cadisAPP

License requirement: TH-PCS

Partial shipments can now be created via the cadisAPP.

“Partial shipments” are a very useful way of taking care of split shipments. Whenever a shipment has been partially loaded our of a depot, but some goods have been left behind the remaining packages need to be transported onwards. The “Partial shipments” feature makes it possible to do this in a controlled and planned manner by creating a new shipment with only the packages that were left behind. (see What is a Partial Shipment? – CADIS Online Help)

see: Create partial shipment via cadisAPP – CADIS Online Help for details on this new feature.

cadisCONNECT (BETA)

cadisCONNECT provides an API layer for programmatic access to cadis functionality.

Depending on the use case, APIs may be designed to

  • integration use cases: retrieve detailed information about specific business objects (e.g., the ETA for the delivery of a certain shipment) or perform actions in cadis (e.g., post a note to a shipment)

  • Mass data load: perform cyclic data retrieval for use in business intelligence solutions.

The API documentation for the current product major release is available via https://demo.cadis.cloud/cadis-rest-public-doc/ with a “Swagger UI” fronted. The URL is protected by a login. If you are interested in cadisCONNECT please contact your project-, service- or account-manager for login data and a brief introduction by one of our consultants.

The API documentation for your current cadis release version is also directly available on you own cadis domain under the same path (https://<your.cadis.domain>/cadis-rest-public-doc/). Please sign into cadisSTUDIO first to authenticate before accessing the documentation URL. Please note that all API specs can be read on the public documentation, but to actually use the API requires the appropriate license to be activated and for the user/token to have the relevant permissions assigned.

Please note that we are currently in BETA with active development ongoing.

  • available APIs don’t yet cover the full range of cadis

  • APIs are subject to change

  • we appreciate feedback!

Appendix 1: Interface updates

This chapter covers changes to the message-based cadis interfaces as well as new message-based interfaces. Please check the sections on “cadisCONNECT” for updates to the cadis APIs.

cTVehicleTypes

An element of type “LanguageStrings” has been added for managing the translated “Vehicle Type Name”.

If the optional element is omitted, the previous behaviour is maintained (i.e. “VehicleTypeName” is used). In case both “VehicleTypeName” and “VehicleTypeNameText” are filled the latter takes precedent.

Appendix 2: New run time configuration options

Please contact our service organization to arrange for a consulting meeting to improve your workflows by leveraging these new options.

Availability and scope of configuration options depends on the underlying functionality being licensed.

New/Changed PUD Run Time Configs

  • Use conventional routing for sequencing out of bounds (PTV2)
    If enabled a trip sequencing request with locations outside the configured bounds will fall back to standard routing without PTV feature layer and no HPR

  • Background sequence optimization mode after invalidation
    If a mode is set, a background sequence optimization (with default parameters) will be enqueued for trips that received an invalid optimization state. If the optimization state is already invalid before any change and a new relevant change occurs the optimization will only be triggered if it’s a user-based action (like dispatching).

  • Sub status mapping by package status
    Depending on the status of the packages in a collection or delivery, it is defined here which sub-status is set for the order in case of an order status calculation. Each category can result in an individual sub-status.

    • Available categories for delivery:

      • all packages delivered with deviation

      • at least one package is marked with an exception or deviation, but the remaining were delivered successfully

      • at least one package was not delivered (refused) and at least one package was delivered with or without deviation/exception

      • at least one package is missing, and at least one package was delivered with or without deviation/exception

      • at least one package is missing, and all other packages were not delivered (refused)

      • all packages refused

      • all packages missing

    • Available categories for collection:

      • all packages picked up with deviation

      • at least one package is marked with an exception or deviation, but the remaining were picked up successfully

      • at least one package was not picked up (refused) and at least one package was picked up with or without deviation/exception

      • at least one package is missing, and at least one package was picked up with or without deviation/exception

      • at least one package is missing, and all other packages were not picked up (refused)

      • all packages refused

      • all packages missing

  • Display loading equipment confirmation only in the first operation
    Is applied to all processes in the terminal (unloading, inventory, loading): If loading equipment processing is mandatory for shipments with exchangeable loading equipment, the loading equipment dialog in cadisApp is only displayed automatically in the first process in which the shipment is scanned.
    Example: A shipment X is unloaded from unloading list A and loaded onto loading list B. An inventory takes place in between.
    1. shipment X is physically unloaded, but not scanned (and of course no loading equipment is recorded).
    2. shipment X is scanned for the first time during the inventory. The loading equipment entry screen is displayed, as this is the first process. The user enters the loading equipment.
    3. The shipment is scanned in loading list B: The loading equipment screen is not displayed because the loading equipment has already been recorded in the inventory.
    Variant: Step 2 is omitted. No inventory has taken place. Loading is now the first process. The loading equipment entry screen is displayed.

  • Do not export 0 amount for actual loading equipment in unloading/inventory/loading status
    In the status exports for terminal handling processes, the export for “0” actual loading equipment (Amount) can be deactivated in the LoadingEquipmentStatus tag.
