New functions in cadis Release 4.18

Logistic Process Platform (LPP) – General functionality

If not stated otherwise new features are generally available in all basic service cut levels (LPP-ENT or LPP Cluster) but accessibility or utility may vary with the limitations of the service cut and your individual usage of cadis. In some cases, availability of information and/or functions in a feature may depend on service cut and other licensed options.

New cadis Online Help

The cadis online help hast moved to the Internet at https://docs.cadis.cloud

Here you will find the established “User manual” (under “For Users”) and completely new materials including “How-To Articles and FAQs” and Training materials like videos or presentations.

In addition, there are areas for Product information (Release notes, Product descriptions etc.) and technical resources (API documentation, admin guides, whitepapers).

Use the search function and explore the new material. Make sure to come back regularly as we are providing new and updated material all the time.

You can switch the language to your preference from the menu. While the user manual is always kept up to date in DE, EN and FR, some articles in other sections, especially multimedia materials, are not available in all languages all the time. It can be worthwhile to browse these sections in any additional language you feel comfortable with.
(Technical resources targeted to developers, IT administrators and architects are provided exclusively in English. We have found that translating highly technical descriptions is prone to causing misunderstandings resulting in misconfigurations and bugs.)

Access to the cadis help is restricted by a login. To access the information, you will need either a user login or, more commonly, a special URL containing an automatic login token.

The easiest way to access the online help is to press “F1” or choose “Help” from the menu in either cadisCONTROL or cadisSTUDIO. You will be automatically redirected to the docs.cadis.cloud website and logged in as an anonymous “cadis user”.

A “cadis user” has access to the “User Resources” section containing the user manual, How-To/FAQ articles and training materials.

If you need access to the extended “Product Information” or “Technical Resources” we will provide you with a special URL that contains a token. Using this URL will automatically log you in as a more privileged “cadis customer” user. If you are reading this online you will already have received and used such an URL. Be sure to save the full URL as you will need it to come back here after you are logged out of the cadis online help. (Hint: You will also need it if you use “F1” in cadis as described above. Doing so will switch you to the “cadis user” login with more limited access.)

cadis Message Channels

License Requirement: LPP-MSG

cadis has added a new module for advanced group communication “Message Channels”.

“Message Channels” currently supports the following channel types:

  • User – Supervisors: communication between an individual user and a group of supervisors (i.e. dispatchers with a terminal worker, admins staff etc.)

  • Terminal – Supervisors: Group communication between all terminal staff and supervisors

  • Vehicle – Supervisor *): the long-established vehicle messaging is fully integrated into the “Message Channels” module.

*) Vehicle channels remain available to PUD customers without additional licensing. They do not yet support sending images

Note: cadisAPP on Android requires Chrome version 71 (or Android the corresponding WebView) to visualize the Message Channels.

Please see “How to administrate message channels” and “How to communicate via message channels” for details on set-up and usage.

Improved Dangerous Goods Display in cadisSTUDIO shipment details

License Requirement: PUD or TH

The display of dangerous goods information in the cadisSTUDIO shipment details and cadisAPP “InfoScan” has been improved.

The main dangerous goods overview now not only displays the hazard label but additional information such as UN-Number, substance, and the dangerous goods points. The hazard labels themselves have been update to the hazard class specific versions.

Click on the dangerous goods card to get a screen with all available details.

Columns in Mobile Accesses Overview

The columns displayed in the tab “Mobile Accesses” have been altered to avoid confusion: “Provided” (printed or emailed) and “Last used” (used to register mobile device) have been removed, instead “last communication” is displayed showing the time a mobile device has communicated last. This makes it much easier to determine at a glance if a mobile registration is being actively used.

The removed values are still available in the details of the mobile access record.

Custom Shipment Attributes

Shipment data can now be extended with custom shipment attributes via configuration. The host system can add those configured attributes to the shipment data exported to cadis via cTExpectedDepotItems (or any interface where the “cTItem” structure is used to send shipment data). The new attributes are displayed in the cadisCONTROL shipment overview, and they are available in as conditions in the rule definitions.

Trip Sequence Control for Imported Trips

License Requirement: PUD-DRV, PUD-DPL, (PUD-TRO for sequence optimization)

It is now possible to control the sequence of trips imported from the host system (cTTour) in cadis. In this mode the content of a trip (the “run sheet”) is controlled by the host system but the sequence of the stops can be adjusted in cadis.

In this mode only the initial trip sequence is taken from the first cTTour import (the CREATE event). Subsequent updates of the trip from the host system (UPDATE events) will no longer affect he sequence in cadis so as not to overwrite potential changes.

It is also possible to calculate the optimal stop sequence and PTAs (Planned Times of Arrival at the stops) using trip sequence optimization in cadis (license PUD-TRO). 

Automatic Trip Sequence Optimization for Imported Trips

License Requirement: PUD-DRV, PUD-DPL, PUD-TRO

It is now possible to automatically calculate the optimal stop sequence and PTAs (Planned Times of Arrival at the stops) for trips submitted from the host system (cTTour). In this mode the content of a trip (the “run sheet”) is controlled by the host system but the sequence of the stops is controlled by cadis. It is also possible to keep the sequence from the host fixed and only calculate PTAs.

When automatic calculation is active cadis will try to determine the optimal trip sequence and PTA directly on import of a cTTour.

Notes:

  • this will introduce a slight delay between import of the trip and forwarding to the mobile device while the sequence is being calculated

  • sequence calculation may fail, e.g. because addresses can’t be geocoded, or the trip is impossible to execute within the given restraints. In this case the unoptimized trip will be forwarded to the mobile devices in the sequence received from the host and without calculated PTAs. The failed calculation is visible to the dispatcher in cadis who can take action as needed.

Automatic release of pure collection trip for execution

License requirement: PUD-PLD or PUD-LDR

The standard lifecycle of a trip in cadis requires the user to release a trip for loading and then close the loading for the trip to be released for departure (i.e. the pickup and delivery orders to be sent to the cadisAPP)

This step may be considered unnecessary for trips with no shipments to load at the depot. cadis can now be configured to allow going from “Created” to “Ready for trip start” if there are only pick-ups or direct transports dispatched to the trip. If there are deliveries that need to be loaded the option is not available. A vehicle must be selected for the function to be available.

Please note that there is a slight difference in usage in cadisCONTROL and cadisSTUDIO.

In cadisSTUDIO there will be an additional button in the trips overview labeled “Release trip”. The “Release for loading” button retains its customary function, i.e. releases only for loading.

In cadisCONTROL there is no additional button. If the option is enabled and “Release loading” is used on a trip with only pick-ups and directs “Loading” is skipped and the trip is immediately released for departure. If there are shipments to be loaded, the trip is only released for loading.

The notification can be disabled in the user settings.

Bugfix: Trailer Capacity in Route Optimization and Trip monitoring

The trip monitor overview and the route optimization now take into the capacities of trailers if present. Should you have worked around this issue by adjusting the capacities of the pulling vehicle (truck) to include that of the trailer you may have to revise the vehicle type master data.

Renamed cadisSTUDIO “Workflow configuration”

The cadisSTUDIO module “Workflow configuration” has been renamed to “Rules & Templates” to better represent its actual content.

Added columns to trip templates overview

Multiple columns have been added to the new trip templates overview now showing most of the attributes of the records.

Bugfix: “noEDI” attribute returned to trip templates

In the migration of the trip templates to cadisSTUDIO in R4.17 the “noEDI” attribute was lost. It is now back; please amend you template data as needed.

Bugfix: Strict Enforcement of Mandatory VAS During Stop Finalization

Up to this point it was possible for a user on the cadisAPP to skip the entry of Value-Added Services that were marked as mandatory. Entry is now enforced during the stop finalization workflow. (Please be mindful of potentially incorrect VAS configurations. You may not have noticed that a VAS entry was unintentionally set to “mandatory” in the past as this setting was not fully functional.)

Driver Unloading

License Requirement: TH and PUD-LDR or PUD-PLD

Drivers can now process unloading lists assigned to their vehicle. Users with the permission “entryMDEUnloadingLocalTrafficByDriver” can now perform inbound scanning in the cadisAPP. They are limited to unloading lists assigned to the vehicle for which their cadisAPP is registered (analog to driver loading).

Context Sensitive Handling Instruction Filters

The cadisSTUDIO shipment details section for “Handling Instructions” can now be filtered for the type of handling instructions.

When accessed in the cadisAPP the handling instruction filter will automatically be set to the appropriate context, e.g. when coming from a local traffic loading process the handling instructions will be filtered to the types relevant for local traffic loading (General, Last Mile Loading, Scanning). Users can see the categories with pending instructions and can change the filters to see more instructions.

The handling instructions list now also supports sorting.

Minor changes and notable bugfixes

Please note that not all bug fixes are listed here but only those that may require action.

Bugfix: Scanning at unloading after scanning in inventory generates an error “package already unloaded”

Previous erroneous behaviour: When a package was scanned in unloading after being scanned in inventory cadis refused the unload scan with a message “Package already unloaded” thereby preventing recording of the unloading status.

This has been fixed, the unloading scan will now be accepted with no notification.

Please note that the incorrect behaviour has been present for many years before coming to our attention.

Removed Criteria “Depot status” for cadisAPP Master Pallet Search

The filter criteria “Depot Status” was removed from the master pallet search screen in cadisAPP. This filter caused confusion with users.

Closed Loophole on Loading Equipment Deviation Approval

R4.16 introduced a feature with which a confirmation by supervisor barcode could be required in case of loading equipment discrepancies in loading (R4.16 Release Notes – CADIS Online Help).

There was a loophole by which users could wilfully circumvent this approval by terminating and restarting cadisAPP at a certain step. This loophole has been closed. Any confirmations not executed correctly during the deviation recording will be requested before the loading can be finished on the cadisAPP. The check can be deactivated in the configuration “Loading” with R4.18 LTS 8.

cadisCONNECT (BETA)

cadisCONNECT provides an API layer for programmatic access to cadis functionality.

Depending on the use case, APIs may be designed to

  • integration use cases: retrieve detailed information about specific business objects (e.g., the ETA for the delivery of a certain shipment) or perform actions in cadis (e.g., post a note to a shipment)

  • Mass data load: perform cyclic data retrieval for use in business intelligence solutions.

The API documentation for the current product major release is available via https://demo.cadis.cloud/cadis-rest-public-doc/ with a “Swagger UI” fronted. The URL is protected by a login. If you are interested in cadisCONNECT please contact your project-, service- or account-manager for login data and a brief introduction by one of our consultants.

The API documentation for your current cadis release version is also directly available on you own cadis domain under the same path (https://<your.cadis.domain>/cadis-rest-public-doc/). Please sign into cadisSTUDIO first to authenticate before accessing the documentation URL. Please note that all API specs can be read on the public documentation, but to actually use the API requires the appropriate license to be activated and for the user/token to have the relevant permissions assigned.

Please note that we are currently in BETA with active development ongoing.

  • available APIs don’t yet cover the full range of cadis

  • APIs are subject to change

  • we appreciate feedback!

Product Lifecycle Announcements

Deprecations

We would like to inform you about the deprecation of the following cadis product functionalities.

Deprecated features are still available in the current release, but they may be discontinued soon.

With these notices we would like to give you the opportunity to take any necessary action early and at the most convenient time possible.

  • If you are currently not using the deprecated functionality, please refrain from doing so.

  • If you are currently using the functionality, please plan on migrating to an alternative as soon as possible. Should this be an issue for you please contact so we can work out a migration plan.

You will receive a final “Notice of Discontinuation” at least 3 months ahead of the product major release that will ship without the stated functionality.

If you have any questions or need support, please do not hesitate to contact us directly.

Legacy “data load” APIs

The legacy “cadisAnalytics API” in the “Otms” definition is currently being replaced by new “Analytics data load” APIs in the respective functional modules. ​

This change is necessary as it will significantly reduce infrastructure requirements and deployment complexity by removing intermediate data transformation and storage (background: the “data lake” and “data pump” components will be made obsolete).​

Important: The replacements will contain the same data but in most cases the structure will change in an incompatible manner.​

All APIs under the “Otms” definition except “loadingEquipments/order already have received fully equivalent replacements and are deprecated (see info-graphic). Discontinuation of all legacy APIs will occur with R4.20.

JMS Adapter

The JMS adapter used for connecting to legacy “Enterprise Service Bus” integrations is now deprecated and will have to be discontinued in Q2 2024.

This is due to the required Java client class no longer being compatible with new versions of the application server expected to roll out at that point. Upgrade to new application server versions is mandatory for security reasons.

At this point it is expected that cadis R4.20 will ship without JMS support. Expect the notice of discontinuation to be submitted with the R4.19 release notes.

Appendix 1: Interface updates

Error Response for REST Endpoint cTOutboundTransport

When sending a incorrect messages to a cadis REST message endpoint the service will now return a qualified error response.

In case of an error, the response body contains the following (JSON) structure, which provides an exact description of the cause of the error or the element that caused the processing to be aborted.

Restriction: currently only implemented for importing loading lists (cTOutboundTransport)! Will be extended to outer endpoints in the future.

Example:

[
 {
  "applicationComponent":"LDG",
  "applicationContext":"Dispatch",
  "code":"CHECK_ITEM_DISPO_IS_DISPATCHABLE",
  "overridable":false,
  "internId":68323,
  "hostId":"402",
  "message":"Unzulässiger Dispo-Status (NOK); Sendung 402",
  "defaultMessage":"Invalid dispatching status (NOK); Shipment 402"
 }
]

Description of attributes:

  • applicationComponent = Component/business part of application: LDG, ULD, INV, ITEM, PUD, ..

  • applicationContext = Context (Business/Action/Activity) of error: Dispatch, Undispatch, SetStatus, ..

  • code = internal error code – depends on application part

  • overridable = Error has quality as ‘WARNING’ and could be overridden by business/repeated call – if supported by called service (not usable for Import of loading lists)

  • internId = Internal id of object that causes the error

  • hostId = Host id of object that causes the error (if known and available)

  • message = message of error in language of client (from request) in human readable form or default language, if required language not set

  • defaultMessage = message of error in English (default) language in human readable form
