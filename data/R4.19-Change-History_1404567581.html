R4.19

New and changed functions

Logistic and Planning Platform

  • [RD-60] [SaaS] ESB can now be configured to create AWS S3 copies of all inbound and outbound messages.

  • [RD-26] [Logging] Improve firebase error handling: Add the error code from firebase (MessagingErrorCode) to the exception message; For UNREGISTERED (and maybe others?), log an info but do not throw exception (no monitoring); Add an info-servlet gimmick to test (send notification to arbitrary ID)

  • [RD-11] [cadisAPP] Reuse generated client id as long as the app data is not cleaned up or the app is reinstalled.

  • [RD-138] [cadisAPP] cadisAPP should be able to detect operating on a Keyence device (DX series, BTA700) and use the scanner of this device. [R4.18 LTS-5]

  • [RD-61] [Performance] Fix resource generation if plural item is empty.

  • [RD-18] [Performance] Available result state was added to output in importer servlet otherwise output was empty for empty result text.

  • [RD-121] [Performance] Added info-servlet view “Users and roles” for simple user/role assignment search/export.

  • [RD-135] [Performance] cflush: use dedicated runtime exception instead of IllegalStateException for tree violation.

  • [RD-58] [PTV logging] External (gis.log) and internal (short summaries in server log).

  • [RD-134] [PTV] Allow PTV2 Usage with pre calculated standard hprs for a fix set of vehicle profiles.

  • [RD-56] [Stability] Cpush no longer uses a schedule and multicasting for notifications.

  • [RD-51] [cadisSTUDIO] Display a label for the selected view in cadisSTUDIO. The label is only displayed if the default view is not selected.

  • [RD-6] [Temperature] The telematic provider Masternaut can now be integrated in cadis.

Terminal Handling

  • [RD-69] [cadisSTUDIO, cadisCONTROL] Flexible algorithm for stop-consolidation (within a tour). The (correct) grouping of deliveries/pick ups to stops within a tour has multiple benefits: the tour calculation is improved because stop durations are calcualted appropriately, dispatching is easier because there are fewer entries in the stop list and mobile execution for the driver is easier as the list of stops is displayed concisely and the optimized stop workflows are in effect.

  • [RD-84] [cadisSTUDIO] A first step to integrate line haul loading in cadisSTUDIO.

  • [RD-87] [cadisSTUDIO] Dashboard:  Line haul loadings are added to the dashboard in cadisSTUDIO “Crossdock”.

  • [RD-91] [cadisSTUDIO] Cadis now offers the function “Manually (post)geocode addresses in collection templates”.

PUD

  • [RD-102] [Performance] Support PUD attributegroup extension by OTMS config.

  • [RD-109] [Performance] Info-servlet support for OTMS Config attributegroups.

  • [RD-140] [cadisAPP] Handover points can now be canceled by the user who created them.

  • [RD-141] [Configuration] It is now possible to configure more precisely which order sub-status is calculated from the package statuses in the cadisAPP.

  • [RD-16] [cadisCONNECT] DPT item /{branch}/items/details and /{branch}/items/packages/details/byBarcode: New Attribute “isOnTerminal” information for packages in public rest service to get data.
