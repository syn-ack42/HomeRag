Introducing “Change History” Documentation

We believe that you can never have too much information about the software you use to run your business. That’s why we are committed to providing full transparency about all work done on our product. In addition to this regular Release Note, we now offer an always-updated online listing of all changes in cadis via our online help pages, starting with R4.19.

“All changes” truly means everything—from the latest and greatest feature to the most minor bug-fix. Just browse to “Product Info” => “Change History” to review the detailed change history for each release. Due to the length and constant updates, this list is only available in English.

Each cadis release change history has two main segments:

  1. Main Release Segment: Titled with the release identifier (e.g., “R4.21”). This section contains all changes developed in that release, split into “New and Changed Functions” and “Bugfixes.” This section is not updated post-release, except for documentation errors.

  2. Maintenance Segment: This section includes a chapter for each maintenance release of the main release (e.g., “R4.21-LTS-1”). Whenever a new maintenance version is available, all changes in that version are documented here.

Each software change includes the following information:

  1. A unique number (e.g., “[RD-809]”): If you reported a bug or requested a feature, you might have been given this number as our internal reference. It is also useful if you have any questions.

  2. A topic identifier: This labels the change with a general group of topics, providing context.

  3. A list of older cadis versions where this change has been merged: For example, “RD-1015” has been included in older releases, starting with “R4.19 LTS-8” and “R4.20 LTS.1” in their respective release trains. This change is also listed in the “Maintenance” section of those releases.

Please note that the change history typically does not include information on required licenses. It also includes features that are still in alpha or beta stages. Thus, you might find references to features that you haven’t seen in cadis—they may not be included in your license or might be released only to selected alpha or beta testers.

We will continue distributing this Release Note Letter with every cadis release, where we delve deeper into the changes we believe will have the most impact on you. In this letter, we will describe how you can best utilize new cadis features and where you may need to watch for changes in behavior. However, if you want to understand what lies behind the label “bugfixes and performance improvements,” the change history is the place to find it.

Give it a try and check out the R4.21 change history here: R4.21 Change History

We have updated the tables in cadisSTUDIO with many advanced features. See the following article for an overview of new table features:
How to Use the Advanced Table Features in cadisSTUDIO

Default Row Height in cadisSTUDIO Tables

Due to usability requests, the default row height has been set to “small.” If you preferred the previous default of “Medium,” you can change it via the table settings menu (“Gear” icon at the top right of every table).

cadisSTUDIO Table Filters

The filtering system in CadisSTUDIO has been upgraded. See this article for information on how table filters in cadis work:
How do cadisSTUDIO Table Filters Work?

PTV 2 Maps

PTV 2 maps are now available in cadisSTUDIO and cadisCONTROL. OpenStreetMap, PTV 1 (not in cadisSTUDIO), and PTV 2 maps are now available and can be selected via configuration on depot level.

Note: PTV 2 can not be hosted on-premise . It is, however, available as a SaaS service for our on-premise customers.

PTV 2 Authentication Support

PTV 2 integration now supports authentication via HTTP basic authentication. With this security feature, we can now easily offer PTV 2 as a SaaS component to our on-premise customers.

PTV 2 Geocoding Quality Configuration

The minimum acceptable geocoding quality for automatically geocoded addresses (i.e., without user confirmation) can now be configured in greater detail for PTV (individual scores for city, street, and postcode match quality). See Why are some addresses not (automatically) geocoded? for background on how automatic geocoding works in cadis.

Handling of Trip Properties

License requirement: PUD-PLD, PUD-LDR

The handling of trip properties (e.g., long load, dangerous goods) for trip templates (cadisSTUDIO) and trips (cadisCONTROL) has been reworked and made consistent. For both traffic types—local traffic and line haul—all properties can now be changed at any time until loading is closed, even if a vehicle combination has been determined. If a vehicle or relation (for line hauls) is selected, the properties set in that master data record are applied, possibly overwriting previous manual changes.

We also fixed some minor bugs related to these properties, such as not applying all properties when selecting a vehicle type instead of an actual vehicle.

Notably, a bug-fix for imported loading lists (cTOutboundTransport) now correctly derives the max. allowed gross weight from vehicle master data during import. Previously, this was applied when a trip was modified in cadisCONTROL using the details screen. If you notice changes in behavior related to dispatching checks (i.e., warnings that didn’t appear before), you may want to check your vehicle type master data as these settings may now be applied correctly.

Classification Code in Dangerous Goods Attachment

We added a field for “Classification Code” in the dangerous goods print document, displayed only for substances in dangerous goods class 1. (This change has also been released in maintenance versions of older cadis releases.)

New Integration API for Documented Stops and Order States

License requirement: PUD-CON

We have added two new public API endpoints to request documented stops (optionally including the status of the contained orders) and order states without stop context.

  • Stop Level Information: Provided by the “documentedStops” endpoint, including actual GPS position, user identification, geo-fence timestamps, etc. This information only exists if the stop was documented by a mobile device; it does not exist if order status was set via the office user interface.

  • Order Level Status Information: Provided by the “orderStates” endpoint, independent of documentation by a mobile device.

Other Notable Improvements and Bugfixes

  • Removed unintended Handling Instruction types from runsheet print document. Only instructions of type “General” and “Delivery” will now be printed.

  • cadisSTUDIO collection template: The “Template name” field is no longer mandatory to maintain backward compatibility with templates created in cadisCONTROL, where it was never mandatory.

  • Trips on disqualified (DSQ) local traffic outbound routes now remain in the ‘Loading closed’ state instead of ‘Ready for trip start.’

  • Fix for cTTour Import: Fixed an issue where cTTour was not imported if the vehicle type was a semi-truck tractor (“TruckSemi”). If you want to pass semi-trucks via the cTTour interface, which does not have the capability of setting trailers, you must set a “default trailer” in the vehicle master data as a semi-tractor does not have any loading capacity. If the master data is set up accordingly, cadis will automatically add the selected default trailer to the trip.

  • Proof of Delivery Report: Added configuration for geolocation and GEO-coordinates in the PDF.

  • Proof of Delivery Report: Added configuration for Global Tracking Number and column on the document.

  • The screen to add shipment via MyTrips now only provides dispatchable shipments. This avoids confusion where a shipment has one (or many) associated partial shipments, only one of which can ever be dispatchable at one time.

Remove Expected Storage Position via GUI

License requirement: TH-PCS or TH-LDS

An “Expected Storage Location” that has been assigned via the GUI (cadisCONTROL) can now be removed. The expected location will revert to the value set via the data interface or the rule engine (if any).

Configuration of Loading Meter Recording (cadisAPP Workflow)

The dialog to record loading meters can now be configured to allow the entry of “Free Loading Meters,” “Used Loading Meters,” and “Loading Unit Full” in all variations.

Notable Minor Changes and Bugfixes

  • Temperature monitoring will now be stopped when unloading a package without defining an actual storage position.

  • The minimum number of seals required before closing a loading can now be configured. This setting applies to cadisCONTROL and cadisSTUDIO. The default is 1 for line haul and 0 for local traffic, to maintain previous behavior.

cadisCONNECT

cadisCONNECT provides an API layer for programmatic access to cadis functionality.

Depending on the use case, APIs may be designed to

integration use cases: retrieve detailed information about specific business objects (e.g., the ETA for the delivery of a certain shipment) or perform actions in cadis (e.g., post a note to a shipment)

Mass data load: perform cyclic data retrieval for use in business intelligence solutions.

The API documentation in Swagger format can be downloaded via cadisCONNECT API in OAS3 (Swagger) format from here: cadisCONNECT APIs If you are interested in cadisCONNECT please contact your project-, service- or account-manager for login data and a brief introduction by one of our consultants.

The API documentation for your current cadis release version is also directly available on you own cadis domain under the same path (https:///cadis-rest-public-doc/). Please sign into cadisSTUDIO first to authenticate before accessing the documentation URL. Please note that all API specs can be read on the public documentation, but to actually use the API requires the appropriate license to be activated and for the user/token to have the relevant permissions assigned.

Please note that we are currently in BETA with active development ongoing.

  • available APIs don’t yet cover the full range of cadis

  • APIs are subject to change

  • we appreciate feedback!

Appendix 1: Interface updates

This chapter covers changes to the message-based cadis interfaces as well as new message-based interfaces. Please check the sections on “cadisCONNECT” for updates to the cadis APIs

HostId in item:PackageIdentification

The “HostId” in item:PackageIdentification is now optional for ctItem 2.0 imports. With this change, it is now recommended not to include the HostId for packages unless you need it and are set up to handle packages as individual business objects.
Background: Unintended “swapping” of HostIds has a tendency to cause integration problems. Such “swapping” usually occurs due to the HostId being generated inconsistently by a data mapping. If you need to set a package HostId but don’t have a unique technical identifier available in your source data, using the “DefaultBarcode” is the recommended best practice. (Do not use a loop counter.)

Export of Loading Equipment Amount in cTLoading/Unloading/Inventory Status

The business logic behind the export of actual (recorded) loading equipment amounts during unloading, loading, and inventory, in cases where no entry was made by a user on cadisAPP, can now be controlled via configuration.
We introduce a new configuration: disableExportOfZeroOrEmptyLoadingEquipment. If set to “true,” cadis will not export an amount for a loading equipment exchange if no loading equipment of this type was recorded in the loading/unloading/inventory process.
The default setting is “false” (current behavior). In this setting, if a shipment carries an expected amount of exchangeable loading equipment and the user does not record an amount during the handling process, the loading equipment type is exported with an actual amount of “0” in the corresponding status message.
