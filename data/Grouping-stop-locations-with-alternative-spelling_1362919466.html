Overview

Stops are created in cadis based on the geo-coordinate and the address. If shipments with addresses with different spellings are imported from the host system, this leads to several stops in one trip. Alternative spellings of the name can be maintained in the master data for the stop locations. If, for example, shipments are sent to different departments of a company, only one stop should be created, because the address of the company is always the same.

An alternative name can be entered directly in the stop location details.

It is also possible, that the dispatcher does is out of context of a trip, where the different spelling became apparent.

In the Trip details (doub-click trip) several stops can be marked on the "Stops" tab, to get grouped. The geo-coordinates of the addresses must be the same within the limits of tolerance (3m). I the go-locations are too different the function will not be available.

Because a dispatcher is not necessarily authorized to change the stop location master data, a request for grouping is created first. The request for grouping can then be carried out with the appropriate authorization in the master data.

After the grouping has been confirmed in the stop location master data, it will be applied to all subsequent trips. If a trip has already been crated (including the one from which the "grouping" has been initiated!) the stop consolidation must be recalculated manually by using the "Trip Details -> Edit -> Recalculate Stop Locations" function.

Following use cases for the grouping of addresses are possible:

Use case 1

Several new addresses are imported (different spelling of the names, but same geo-coordinate). An own stop is created for each address within the trip. The user marks the stops and selects "Create stop grouping request" using the context menu.

With existing authorization the user gets to the "Stop location grouping" dialog in cadisSTUDIO.

The user selects an address, for which a new stop location should be created in the master data.

The user confirms with "Create new stop location". The suggested stop location ID is confirmed with "Apply".

The user now sees the new stop location in the master data. The alternative spellings of the name can be found in the stop location details.

(The user generally has the possibility to cancel the request and to carry out at a later date or to delete it â€“ than the request is no longer available and must be restarted if necessary.)

The stops have to be recalculated for the existing trip. Therefore the function "Recalculate stop locations" can be used.

After refreshing, the stops that where selected before are grouped to one stop.

If this use case occurs again, cadis automatically creates only one stop.

Use case 2

There is one address among the imported new addresses, which geo-coordinates already exist as stop location in the master data. All stops are marked and "Create stop grouping request" is selected using the context menu, then the user has no choices in the "Stop location grouping" dialog. The user only can accept to add alternative spellings of the name. The alternative spellings of the name are added to the already existing stop location.

Use case 3

The initial situation is the same as in use case 2, but not all stops are marked, only the new addresses for which there is no stop location in the master data.

Also the non selected (already existing) stop location is listed in the "Stop location grouping" dialog. The user has either the possibility to group the new addresses to a new stop location or to add the new addresses as alternative spellings to the existing one.

Use case 4

Among the imported new addresses are at least two, which have its own stop location in the master data. The user only can take note of this and it is not possible to group anything in this case.

Further information

In the stop location master data there is a button "Grouping requests", which is only visible, if current requests exist.
