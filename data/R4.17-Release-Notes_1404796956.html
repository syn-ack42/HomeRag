New functions in cadis Release 4.17

Logistic Process Platform (LPP) – General functionality

If not stated otherwise new features are generally available in all basic service cut levels (LPP-ENT or LPP Cluster) but accessibility or utility may vary with the limitations of the service cut and your individual usage of cadis. In some cases, availability of information and/or functions in a feature may depend on service cut and other licensed options.

MP process event export

License Requirement: LPP-MP

Event messages can now be emitted from cadis on actions regarding a master pallet. This includes MP lifecycle changes (created, closed, etc.) and adding or removing packages.

Events are emitted as compact json messages on configurable triggers. Please see the corresponding sections in Appendix 1 and 2 of this document (interface updates and new run time configuration).

cadisAPP version schema standardized

The version numbering in cadisAPP is now aligned with other components of cadis.

All public R4 Version numbers follow the schema <major release>.<increment>.<LTS build>

e.g., Version 17.4.1 Refers to the Major Release R4.17, monthly increment 4 (which is the final increment of the major 17 release train and therefore the public release version) and LTS build 1: the first patch build of the major release.

Note:

  • this change causes the app version numbering to change significantly from the previous version, this is not an indicator of a big change it is merely necessary to harmonize the version numbers for easier management

  • this versioning scheme does not apply to customized app versions.

Master data export and import

Most master data types (Vehicle Types, Drivers, Freight carriers, Vehicles, loading container types, loading containers, Last mile routes, Line haul routes, Handling instructions, Terminal locations, Terminal areas) can now be exported and imported via cadisSTUDIO by administrator users. This can e.g., be used to conveniently transfer master data between test and production systems or between org units (branches) on the same cadis instance.

The export will only include master data records defined a manged in the org unit the user is currently logged in to (i.e., not those of records inherited from higher level master data nodes).

When importing the records, they are in turn imported into the org unit the user is currently logged in.

The import is “UPDATE and ADD”. This means that records with the same ID are overwritten, and new records are added but existing records are not deleted.

Easier Customized Texts and Translations

cadis now offers a more lightweight mechanism to adjust the translations of virtually all texts in cadis (cadisSTUDIO, cadisAPP, cadisCONTROL and cadisSERVER) at run time. In addition to the stablished method of managing resource configuration files customized texts can now be managed in the cadis database. These customized texts are not affected by version updates and do not incur service management effort.

Please contact our service organization if you are interested in customized texts and translations.

Mobile License Management

cadis now displays the number of mobile licenses used versus the number of licenses purchased in the “Mobile accesses” screen.

At this point the license statistics are configured for display and review only. It does not yet prevent usage or creation of mobile accesses even if the available license volume is exceeded.

Please note that licenses are purchased per tenant, but the “Mobile accesses” are managed per depot. The “Mobile accesses license usage“ will always show the values for the whole tenant. For this reason, with the exception of “singe depot” tenants, the counter will show more licenses used that there are mobile accesses listed above. The other accesses are simply in another depot on the same tenant. *)

If your license usage count exceeds the licenses available, please delete unused or superfluous mobile accesses. This is a best practice not only for reasons of licensing but also highly recommended to maintain system security. A long forgotten access code can still grant access for a mobile device if it is lost or compromised.

*) There are a few SaaS customers where appropriate tenant separation has yet to be implemented. Your group is treated as a single tenant for purposes of licensing which makes this display irrelevant to you at this time. Please ignore this statistic until full tenant separation has been completed.

Trip Templates migrated to cadisSTUDIO

License Requirement: PUD-PLD (Last mile) TH-PCS (Line haul)

The management of “Trip templates”, aka “regular departures” for local traffic and line hauls has been moved to cadisSTUDIO.

The buttons for “Manage regular departures line haul” and “Manage regular departures local traffic” in cadisCONTROL will redirect to cadisSTUDIO. The “Generate trips from templates” function remains available in cadisCONTROL and cadisSTUDIO.

The user guidance for selecting vehicles and loading containers has been significantly improved in cadisSTUDIO and it is now possible to select either a concrete vehicle from the master data or just a vehicle type in both local traffic and line haul departures.

Order status and Sub status configuration

New status and sub-status combinations for collection and delivery can now be created and changed at run time.

The Status/Sub status texts (including translations) for the cadisAPP and cadisCONTROL/cadisSTUDIO and the colour scheme can be configured.

Note: Workflow and status configuration is very complex and requires intensive training and technical knowledge. As such these options are restricted to certified experts. Please contact our service organization to arrange for a consulting meeting to improve your workflows by leveraging these new options.

cadisAPP callable Phone numbers on cadisAPP

Phone numbers included with the contacts for Consignor, Consignee and Shipper and in the Handling instructions for collection and delivery are now be callable on the cadisAPP directly.

Please note that all telephone numbers must be in a format valid as a callable number in the Android System of the mobile device. This may vary with Android version or regional settings. Otherwise, the number may fail to dial or, in the case of handling instructions, may not be recognized as phone number at all.

BETA: On Tour Hand-Over Points

License Requirement: PUD-HOT

The “On Tour Handover” function allows drivers to deposit goods at a handover point to be picked up by another driver either immediately or at a later time.

The first driver can select delivery stops from his trip and enter another vehicle who should receive them. A “handover point” with these shipments will be created at the driver’s current location.

The driver of the receiving vehicle will see the handover in their cadisAPP where they can confirm the completion of handover. At this point the new stops will be added to their trip.

Select Trip on cadisAPP.

Depending on the “Tip subscription mode” configuration drivers can select trips to work on from the “Trips” menu on the cadis App. The trips can also be removed from the device as long as no stop has been started. By selecting “Details” from the context menu the users can review the details of the trip including the map.

Calculated actual driven distance in “myTrips”.

The actual driven distance on a trip is now displayed in “myTrips” on cadisSTUDIO.

The distance is calculated based on the GPS track of the cadisAPP from the timestamp of the start of the trip until the current time or the end of the trip. Please note that the calculated distance is only approximate and may be negatively impacted by technical issues such as missing GPS signals or by user errors such as closing the loading after the truck has already departed of failing to close the trip (or simply logging out of the App) at the end of the driving shift.

If multiple trips are active at the same time on a vehicle each of those trips will show the full km driven from its individual start and end timestamp (so be careful to not double count distances).

Calculated actual distance in cadisCONNECT

License requirement: CON-TRP and PUD

The actual driven distance on a trip is now available in the cadisCONNECT integration API for trips.

The distance is calculated based on the GPS track of the cadisAPP from the timestamp of the start of the trip until the current time or the end of the trip. Please note that the calculated distance is only approximate and may be negatively impacted by technical issues such as missing GPS signals or by user errors such as closing the loading after the truck has already departed of failing to close the trip (or simply logging out of the App) at the end of the driving shift.

If multiple trips are active at the same time on a vehicle each of those trips will show the full km driven from its individual start and end timestamp (so be careful to not double count distances).

Setting Tour sequence on map in “myTrips”

License Requirement: PUD-PLD or option PUD-DPL, (DPT-TMO-ADV for usage in cadisAPP)

Users of “myTrips” on the browser and in the cadisAPP can now change the sequence of stop on the map.

To change the sequence, activate the “Dispatch” mode (hand symbol), tap/click the stop you want to move then tap/click the trip segment you want to move the stop to.

“myTrips” Filter for Trip Lifecyle

License Requirement: PUD-PLD or option PUD-DPL (DPT-TMO for usage in cadisAPP)

“myTrips” now includes trips during “Planning” and “Execution” phase. You can toggle between trips in Planning stage an Execution stage via the lifecycle filter on the top left in both the desktop and mobile views. A trip is considered in “Execution” after closing of the loading.

myTrips Dispatching: Control Sequence of Dispatched Collections

License Requirement: DPT-TMO-ADV, PUD-PLD or option PUD-DPL

Users can now select if pick-up and directs shipment added with the cadisSTUDIO “Add shipment (to trip)” dialog are added at the end of the trip (current default) or will be kept in between deliveries. To achieve the latter behaviour the checkbox “Use sequence of added shipments as sequence of trip“ checkbox must be selected.

The default setting of this checkbox can be controlled by configuration.

If enabled via config the handling mode (standard, dynamic, noEDI) of unloading lists can be changed via cadisCONTROL. Master Pallet in Info Scan

License Requirement: LPP-MP, DPT-ISC-ADV (for actions in info-scan)

Display and handling of the master pallet in the info scan has been greatly improved. The info scan on a master pallet now also displays MP-specific content like the MP type and content of the MP.

BETA: Local Traffic Loading in cadisSTUDIO

Users can now monitor and manage local traffic loading activity on cadisSTUDIO. The current functionality includes a dashboard with live statistics of current local traffic loading lists and scheduled shipment volumes and the “My Loadings” screen with all current loadings and details with the capability to release and close loading lists.

The functionality is fully responsive for mobile devices so loadings can be managed directly on the ground on the cross dock from tablet or even a smartphone.

Other functionality currently available in cadisCONTROL like setting/overruling loading status, management of line haul loading lists, adding and removing shipments, creating loading lists, etc. will be added in the next few major releases. At this point the cadisSTUDIO features are complementary to the established cadisCONTROL functionality. Users are free to use either or both in parallel according to their preference or the task at hand.

Improved Handling of Main- and Partial-Shipments in Unloading

There are cases in which a user can create a situation where both a partial shipment and the corresponding main shipment are added to an unloading list. This can happen if a partial shipment containing packages “c” is expected on a loading list and the user scans a package “b” of the corresponding main shipment and confirms unloading of this package on the unloading list. cadis used to show both the main and the partial shipment with all their packages, i.e., it used to show the partial shipment with package “c” and the main shipment with packages “a”, “b” and “c”. As a result, package “c” was shown and counted twice.

Now cadis will only show those packages under the main shipment that are not already included in the partial shipment.

The mandatory photo and the minimum and maximum number of photos required on scanning an unknown package (barcode with no shipment data in cadis) in unloading and inventory can now be deactivated via configuration.

Changed wording “Wrong Scan”

A non-effective scan (e.g., a loading scan that was forbidden; a scan of a surplus package in unloading that was cancelled by the user …) used to be called “Wrong Scan”. These scans are now called “Scanned but not loaded”, “Scanned but not unloaded” etc. in the package event history by default.

Note: all texts and translations in cadis are fully configurable so your wording may have been changed in the past.

cadisCONNECT (BETA)

cadisCONNECT provides an API layer for programmatic access to cadis functionality.

Depending on the use case, APIs may be designed to

  • integration use cases: retrieve detailed information about specific business objects (e.g., the ETA for the delivery of a certain shipment) or perform actions in cadis (e.g., post a note to a shipment)

  • Mass data load: perform cyclic data retrieval for use in business intelligence solutions.

The API documentation for the current product major release is available via https://demo.cadis.cloud/cadis-rest-public-doc/ with a "Swagger UI" fronted. The URL is protected by a login. If you are interested in cadisCONNECT please contact your project-, service- or account-manager for login data and a brief introduction by one of our consultants.

The API documentation for your current cadis release version is also directly available on you own cadis domain under the same path (https://<your.cadis.domain>/cadis-rest-public-doc/). Please sign into cadisSTUDIO first to authenticate before accessing the documentation URL. Please note that all API specs can be read on the public documentation, but to actually use the API requires the appropriate license to be activated and for the user/token to have the relevant permissions assigned.

Please note that we are currently in BETA with active development ongoing.

  • available APIs don’t yet cover the full range of cadis

  • APIs are subject to change

  • we appreciate feedback!

Legacy “data load” APIs

The legacy “cadisAnalytics API” in the “Otms” definition is currently being replaced by new “Analytics data load” APIs in the respective functional modules. ​

This change is necessary as it will significantly reduce infrastructure requirements and deployment complexity by removing intermediate data transformation and storage (background: the “data lake” and “data pump” components will be made obsolete).​

Important: The replacements will contain the same data but in most cases the structure will change in an incompatible manner.​

Most APIs under the “Otms” definition already have received fully equivalent replacements and are deprecated (see info-graphic). The other APIs will be deprecated in R4.18 with the introduction of fully equivalent replacements. Discontinuation of all legacy APIs will occur with R4.19.

Appendix 1: Interface updates

This chapter covers changes to the message-based cadis interfaces as well as new message-based interfaces. Please check the sections on “cadisCONNECT” for updates to the cadis APIs.

New Master Pallet Event interface

This Interface provides two message types (in json format):

  • Process event

  • Package event

Process events are triggered when a master pallet changes its lifecycle status (created, closed, etc.) according to the trigger configuration.

Package events (if enabled in configuration) are triggered when packages are added or removed from the MP.

Note that event messages are kept small. If details of the master pallet are required for processing an event cadisCONNECT provides a corresponding REST GET interface to retrieve such data.

The attached documentation is provided in “asynapi” documentation standard which is very similar to the OAS3 (“Swagger”) standard but extended to allow documentation of event channels. Documentation readers (similar to “SwaggerUI”) are available freely for download or online use e.g., here: https://studio.asyncapi.com/

See mapaEvents_R4.17.yaml
