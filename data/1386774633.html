General

The feature „Arrival Notification“ serves the purpose to notify / contact consignees or involved persons of a shipment:

  • To inform them about the expected time of arrival. Among other things it can be ensured, that the affected person can be met at this time to receive the shipment personally respectively that the delivery / collection will be successful.

  • To discuss further details regarding the collection / delivery.

Therefore the driver has to be made aware of that there exist orders, which require to notify the consignee:

  • At runsheet confirmation

  • xx minutes before arrival (depends on PTA / driver-ETA)

  • for the next x stops, if there are still open arrival notifications

The notification is usually (but not only) done by the driver and happens before the arrival at the delivery-/collection stop.

Depending on the configuration the dispatcher can also trigger an automatic notification. The corresponding channel has to be connected in the project. At the moment only phone call and SMS-sending are possible.

Delimitation of terms

PTA

Planned time of arrival (also for cadisAPP) – determined by cadis during the planning process, for example by the stop sequence optimization. It is relevant as fallback, if no driver-ETA is available.

ETA

Estimated time of arrival at the (unprocessed) stop (also for cadisAPP), which is constantly being updated and determined during the trip either with PTV depending on the received GPS position or without PTV depending on the reported status values.

Driver-ETA

An arrival time that is estimated / entered by the driver at the (unprocessed) stop. This ETA is completely independent of the automatically calculated ETA. The driver ETA, if available, will be used for arrival notification.

Arrival notification / Driver notification:

The consignee / shipper will be contacted by the driver during the tour execution (to inform the consignee / shipper when the arrival will be or to discuss other details).

Configuration options

There are various configuration options available for the arrival notification.

Data input / Activation of the arrival notification

Whether an arrival notification needs to be carried out for a shipment can be determined either via item import or by editing a shipment.

Import

The currently implemented cTItem-2-0-interface is used.

The element isAvisDriver (part of TypeAttributes, Boolean, optional) indicates, if an arrival notification has to be carried out for a shipment. The person to be contacted is transferred as InvolvedPerson of type DriverPreadvContact.

Editing a shipment

The attribute arrival notification (cadisCONTROL) respectively driver notification (cadisSTUDIO) (is equivalent to the element isAvisDriver in the import) can be (de-)activated for (regular) collections and shipments via the attributes of the shipment details (corresponding permissions are required).

View in cadisSTUDIO

Update behavior

When the isAvisDriver flag is updated (whether by the host or manually), no update of the PUD orders is triggered and therefore the changes are not transferred to the MDE. The data only comes to the PUD order / to the MDE when the trip export is triggered by any other changes of the tour or one of its items.

A change of the contact person of type DriverPreadvContact causes an immediate update of the order in PUD and on the MDE.  

Processes in cadisCONTROL

Display of the arrival notification

In cadisCONTROL, a symbol is used to indicate whether an arrival notification needs to be carried out for a shipment:

open; the arrival notification has still to be carried out

done

The symbol is displayed

  • In the shipment details

  • In the shipment overview

  • In the trip monitor (triplevel, stoplevel, map)

  • In the order details

Information about the execution of the arrival notification can also be found in the history of the shipment and order details.

Execution of the arrival notification in cadisCONTROL

The dispatcher can document the arrival notification for shipments / collections via the GUI in order to be able to process the arrival notification even without a driver.

Following functions are available, if configured:

  • Document the execution of the arrival notification on the shipment (Configuration parameter: CadisArrivalAvisConfiguration.arrivalNotificationAllowedForGuiUser)

  • Trigger the automatic notification by an connected system (Configuration parameter: CadisArrivalAvisConfiguration.serversideArrivalNotificationSupported) 

Preconditions

The function is accessible, if:

  • It is a collection or a shipment (no direct shipment)

  • The shipment has the attribute “Arrival notification”

  • The shipment is dispatched and the trip has at least the status trip released

  • The configuration parameter Arrival notification by GUI user is activated.

Special user permissions are not necessary.

Process

The arrival notification can be executed from the context menu of a shipment / collection in the shipment overview or from the shipment details.

A window opens. It shows the relevant shipment data, which could be necessary for the notification:

  • Relevant shipment data

  • PTA / ETA of the associated stop, if available

  • Contact person and contact information according to InvolvedPerson of type DriverPreadvCont.

It can be chosen between documenting an arrival notification (which may be carried out manually by the dispatcher) or automatic notification by the server (if configured). The arrival notification carried out will always be documented.

The name of the contact person can also be entered (optionally). Once done, the mask closes.

As a result of the notification the status of the shipment changes to “done” and the execution can be traced in the shipment history and in the order history (PUD).

The driver is not informed about this action.

Processes in cadisAPP

Display

In the cadisAPP following icons are used to represent the arrival notification (on trip level, stop level and order level):

Open (Red telephone receiver, filled out)

Done (Telephone receiver with gray border, not filled out)

After the loading scan the icon is shown for a shipment.

Trip level

Is trip confirmation configured, the driver will be made aware of the number of shipments in the trip that require arrival notification.

In addition, the driver is guided through a list of all arrival notifications to be made, in which he can also see the existing information on the planned / expected arrival time for the shipments. Shipments without a planned / expected arrival time will be highlighted separately.

Whether the driver can, must or may enter a driver ETA for shipments without a PTA / ETA when confirming the trip is controlled by the configuration parameter CadisTourConfirmationConfiguration.arrivalAvisValidationOption.

Stop level

Filter options

The stop list can be filtered for stops

  • With arrival notification (independent of the status)

  • For which no PTA neither driver ETA is available (unknown arrival time)

There are corresponding entries in the filter in the “Other” section.

Functions

Push notification

It is checked cyclically whether an arrival notification is required for stops/orders. If this is the case, the driver will be alerted to upcoming arrival notifications via push notification.

This happens either for a specific period of time in the future (arrival notifications due in the next x minutes), or for the next y stops.

The mechanism is configurable via the parameters CadisArrivalAvisConfiguration.preWarningTimeInMinutes and CadisArrivalAvisConfiguration.numberOfStopsToCheckForArrivalAvis.

The dialog for carrying out / editing the arrival notification is reached by clicking on the push notification.

Remember later

The driver can choose to be reminded of the arrival notification again via push notification in 1/15/30/60 minutes (or any input). The arrival notification dialog will then close.

Execute / document the arrival notification

All aspects relating to arrival notification can be handled via the “arrival notification” dialog.

The dialog can be reached

  • By clicking the push notification (shows all shipments with due arrival notifications)

  • Via the order list in the context menu “contact person” of an order (shows all shipments of the associated stop with open arrival notifications)

In the list, the driver sees the orders with open arrival notification, the contact person to be notified and the telephone number and, if applicable, the arrival time. The last last known arrival time is displayed – either the PTA / ETA determined by cadis / Host or the driver ETA). The list can be filtered for open, overdue arrival notifications.

The arrival notification is done per order (shipment). A stop can have multiple orders with the same or different contacts. In this case, multiple notifications would be required for a stop.

The arrival notification can be done directly for each order using the context menu in the list. There are the following ways to edit an arrival notification:

  • Call contact person: displays the relevant contact details. If configured (CadisArrivalAvisConfiguration.callContactByPhoneAllowed) the contact can be called directly from the mask. In addition, the actual contact person who was spoken to can optionally be recorded. The driver then confirms with “Contact person called” that the arrival notification has been carried out.

  • Trigger automatic notification (only available if configured via CadisArrivalAvisConfiguration.serversideArrivalNotificationSupported): This option triggers a connected system to send the arrival notification. This could be SMS, email or similar. The corresponding system must be connected and configured specifically for the project.

No further action is required then; the order disappears from the dialog and the status of the arrival notification (telephone receiver icon) changes to “done” (white telephone receiver).

Record / Edit arrival time (driver ETA)

The driver can record or change the estimated arrival time. This serves as an indication of when the arrival notification is due. The driver ETA is not transmitted to the depot but is only used by the app for display and calculations. The registered driver ETA cannot be in the past. The driver ETA can also be recorded as part of the tour takeover.
