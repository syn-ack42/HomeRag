Introduction

This release note is based on cadis R4 2025.3.0

Please take a look at the R4 2025.3 Change history, which contains the full list of changes.

Many of the features listed here are available across all basic service cut levels. However, some may require an additional licence, while others may have varying accessibility or utility depending on service cut limitations and your specific use of cadis. In some cases, the availability of information and/or functionality within a feature may also depend on the service cut level and other licenced options. 

Highlighted Enhancements

Driver selection in trip template (RD-3330, cadisSTUDIO)

We added the ability to select a driver directly within the trip template dialogue (Local Traffic). This allows dispatchers to override the default vehicle driver and assign a specific person before trip generation.

Driver selection in route details (RD-1980, cadisCONTROL)

Introduced a new driver field in route details for both Local and Line Haul trips. Dispatchers can now explicitly assign a driver from master data, independent of the vehicle. This improves clarity for planning, printing, and mobile access. If a default driver is set in the vehicle data, it is prefilled but can be manually overridden. The selected driver is also included in CTLoadingStatus exports using the "Driver1" role.

cadisCONNECT for Unloading (RD-874, cadisCONNECT)

Two new endpoints — /list and /details — provide structured access to unloading data, mirroring existing loading APIs. Third-party systems, analytics tools, or partners can now query unloading workflows just as easily, paving the way for richer visibility, auditing, and cross-system automation.

Rules engine: Add the Name2 for shippers & consignees (RD-2749)

Extended the rule and parameter definitions to include Name2 (and other secondary fields) for shippers, consignees, pickup, and delivery addresses. This improves flexibility when building rules in industries where secondary naming schemes are often used (franchises, pharma chains). Smarter, more granular rule logic means fewer manual exceptions and more precise auto-dispatching.

Driver Pairing: Update the driver ID in the trip details on log-on (RD-3199)

Enhanced logic for driver-paired mobile access: when a driver logs in and subscribes to a trip, the system automatically populates the trip's driver field. This improves traceability and planning visibility when working with flexible driver-vehicle assignments.

Added additional sequences to LoadingStatusItems in cTLoadingStatus exports (RD-2832)

The cTLoadingStatus export now optionally includes three new fields for each item:

  • PlannedLoadingSequence – reflects the planned loading order in the cadisAPP UI

  • ActualLoadingSequence – reflects the order in which packages were scanned (first scan of first package in item)

  • PlannedStopSequence – reflects the stop position in the trip

These values help downstream systems identify multiple types of sequences. The new fields are enabled via configuration.

The established “Sequence” field remains unchanged.

Improvements

  • RD-1797 – If a driver logs in to cadisAPP using a Driver-type access, their ID is now shown in the trip details. This resolves previous issues where the trip lacked a linked driver unless the vehicle had a default driver.

  • RD-3647 – Added a "Skip" button to the "Geocode all not geocoded shipments" workflow in cadisCONTROL, allowing users to skip individual entries without forcing geocoding.

  • RD-3803 – Removed a non-functional “Generate unloading list“ action in cadisCONTROL for line haul trips (this function is only available for local traffic trips).

  • RD-3722 – Added a new configuration option to automatically release unloading lists created for line haul traffic, enabling seamless host integrations. This avoids manual intervention and supports automation of delivery chains.

  • RD-3654 – Added a new "Driver" column in the Trip Template overview for both Last Mile and Line Haul templates.

  • RD-3651 – Optimised the /unloadings/details cadisCONNECT endpoint now includes coordinates in address fields and removes deprecated master pallet references at item level.

  • RD-3648 – Changed the display unit for package dimensions in the driver app from meters (rounded) to centimetres with one decimal point. This ensures accurate loading decisions and avoids rounding issues that led to user frustration.

  • RD-3546 – Users can now request additional mobile log files even when a previous request is still open. A confirmation dialogue is shown, improving support flexibility and avoiding user lockout in edge cases.

  • RD-3531 – Standardised geocoding status icons in cadisCONTROL to match cadisSTUDIO. The system now clearly distinguishes between auto-geocoded, manually geocoded, failed, and in-progress states using consistent visuals and tooltips.

    Geocoding: Failed - OK/Automatic - OK/Manual - In Progress

  • RD-3427 – Implemented a new push-based temperature data interface for NIC-Place.

  • RD-3199 – Introduced a performance safeguard feature in shipment tables. A configurable cap prevents excessive query loads and guides users to refine filters when too many rows are scanned

  • RD-3946 - In addition to the "Default Driver" (the default driver assigned to the vehicle in the vehicle master data) cadis now also exports the actual driver selected in the trip (see "Driver selection in route details, RD-1980"). The actual driver is exported in the "InvolvedPersons" structure as an "InvolvedPerson" with "InvolvedPerson"="Driver1" 

Bugfixes

  • RD-3421 – Vehicle table now defaults to the OrgUnit’s local date instead of UTC for daily filtering.

  • RD-3503 – “Departures” column now correctly displays values in the Vehicles table.

  • RD-3399, RD-3477, RD-3480, RD-3482 – cadisAPP: Fixed some rare crashes and data sync issues

  • RD-3789 – Fixed a bug where tour stop sequences could become corrupted during mass item updates and drag-and-drop assignments in the tour monitor when invalid geo-positions were present.

  • RD-2639 – Stops now continue to appear in MyTrips even after an order status is set manually in cadisCONTROL. This resolves inconsistencies where stops seemed to disappear prematurely.

  • RD-3777 – Resolved a bug where arrival timestamps were incorrect after cadisAPP reboots.

  • RD-3683 – Fixed sorting issues in the Trip Template view (Vehicle, Created, and Last Changed)

  • RD-3680 – Fixed an issue where cadis-generated unloading lists failed to be created when containing multiple partial shipments for a single main shipment. The import logic now merges such entries properly to prevent schema violations.

  • RD-3655 – “Driver” field in Local Traffic trip details is no longer pre-filled with the label "Default Driver" if no such driver exists, preventing confusion for planners during vehicle changes or manual assignment.

  • RD-3651 – Optimised the /unloadings/details cadisCONNECT endpoint to prevent request failures when too many unloading IDs are queried at once.

  • RD-3642 – Fixed a bug where trips could be cancelled in the UI even if “To be unloaded” shipments remained. The system now prevents cancellation ensuring consistent trip handling.

  • RD-3483 – Fixed a bug where rejecting a tour triggered a misleading error message. The app now correctly transitions to the stop overview without false alarms or repeated prompts.

  • RD-3305 – Corrected unloading list behaviour for collections. If a previously scanned collection is marked as “Not Collected” and then corrected back to “Collected OK,” it is once again included in the unloading list.

  • RD-2702 – Fixed an issue where when unknown packages were later merged into known shipments (e.g. after import) previously documented actual loading equipment data was not retained and assigned to the resulting shipment. This fix prevents data loss during merge operations.
